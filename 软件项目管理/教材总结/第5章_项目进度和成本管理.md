# 第5章 项目进度和成本管理
> [!NOTE] 本章主线
> 本章把计划落到时间和费用控制上，复习时按“**==活动识别==** → **==活动排序==** → **==关键路径==** → **==里程碑==** → **==进度编制==** → **==进度/成本控制==**”理解。
> 重点是**==关键路径==**、**==网络模型==**、**==里程碑==**和**==偏差控制==**的计算与判断。


## 一、按教材顺序保留的必备概念原话

### 5.1 标识项目活动

- 项目进度、成本管理的主要工作包括项目活动的标识、活动的排序、活动的资源估算、活动的成本估算、活动的工期估算、资源合理分配、制定项目完整的进度和成本计划、监控和控制项目进度及成本等相关的内容。
- 标识项目活动就是把项目的工作量分解为易管理的具体任务，而每一项任务都要有明确的时间和资源的限制，它是项目进度表编制的基础。
- 全面、清晰、详细地定义所有项目活动，可以从两条主线考虑：
  1. 软件开发生命周期：以软件开发周期模型的各个阶段对项目进行阶段性的划分，再结合软件项目的需求详细地考虑每个阶段的活动。
  2. 软件开发功能点：以软件项目的需求分析为主线，对软件需求进行分析和整理使其形成各个功能点模块，然后结合软件开发周期对各个功能点模块进行细分。
- 标识软件项目活动时，最后分解的项目活动应该是明确的、可管理的和可定量检查的。
- 活动分解的常用方法：逐层分解、使用模板、专家评定、滚动式规划。
- 滚动式规划是一种逐步完善的规划方式，对近期完成的工作进行细致规划，而对远期完成的工作进行初步规划，是将计划期不断向前延伸、连续编制计划的方法。
- 在项目活动标识过程中，不仅要把所有活动定义出来，还要分析活动的两种类型：前导活动和后续活动，即各活动之间的相互依赖关系。
- 前导活动是在下一个活动之前必须出现的活动；后续活动是在前导活动之后必须出现的活动。
- 活动标识结束后，应形成明确的活动清单，包括活动定义、类型分析以及相互依赖关系。

### 5.2 确定项目活动的次序

- 项目活动之间的依赖关系是指活动在时间上的逻辑顺序。活动之间的依赖关系取决于实际工作的要求，不同活动之间的依赖关系决定了活动的优先顺序及其重要性。
- 活动关系类型：

| 关系类型 | 缩写 | 教材要点 |
| :--- | :---: | :--- |
| 结束-开始 | FS | 活动 A 结束后活动 B 开始；这是最普遍也是最常用的活动类型。 |
| 开始-开始 | SS | 活动 A 开始后，活动 B 才能开始；经常表示并行且具有一定依赖关系的活动。 |
| 结束-结束 | FF | 活动 A 结束后，活动 B 才结束；经常表示并行但产出物具有一定依赖关系的活动。 |

- 重叠时间也叫超前时间（Lead Time），在关系顺序上允许提前后续活动的时间。
- 滞后时间（Lag Time）是在关系顺序上允许推迟后续活动的时间，即活动之间拖后或等待的时间。
- 项目网络图是显示活动顺序的首选方法。创建项目网络图常用前导图法和箭线图法。
- 前导图法（PDM）又叫单节点网络图法（AON），用单个节点表示一项活动，用节点之间的箭线表示项目活动之间的相互依赖关系。
- 箭线图法（ADM）又叫双代号网络图法（AOA），用箭线表示活动，活动之间用节点连接，只能表示结束-开始关系，每个活动必须用唯一的紧前事件和唯一的紧后事件描述。
- 虚拟活动在图形中用虚线箭头表示，不消耗时间和资源，主要用来理顺依赖逻辑。
- 网络图绘制的基本原则：正确表达逻辑关系；不允许循环回路；节点之间严禁出现双向箭头或无箭头连线；只能有一个起始节点和终止节点；不允许孤立节点；箭线应避免交叉；非时间坐标网络图中箭线长短与持续时间无关；箭线方向以从左向右为趋势；布局应条理清楚、结构整齐。
- 网络图完成应伴有简洁说明，描述排序原则、依据和不平常的排序。

### 5.3 关键路径分析

- 关键路径分析是在项目进度管理中应用最为广泛的网络技术之一，它借助网络图和各活动所需时间来估算项目的总工期。
- 关键路径法（CPM）可以为进度管理工作指明方向，为项目经理提供决策依据。
- 在项目网络中会有若干条网络路线，对比各网络路线的累加工期，通常有一条路线的时间最长。这条路线决定项目的工期时间，称为关键路径。位于关键路径上的活动就是关键活动。
- 活动缓冲期（总浮动时间）是指在不导致项目预估工期延迟的情况下，各个活动可以有多少时间的延迟。
- 关键活动的缓冲期都是 0。任何关键活动的延迟都会导致项目工期延期。
- 非关键活动缓冲期的简化计算：找到包含该活动的最长非关键路径，用关键路径时间减去这条路径时间，得到的时间差就是该线路上活动的缓冲期（前提是其他活动不延迟）。
- 压缩工期通常是在保证质量的前提下，寻求任务、时间和成本三者之间的最佳平衡。
- 要使整个项目缩短工期，试图缩短非关键路径上的活动周期没有用；只有使关键路径的工期缩短，整个项目才可以提前结束。
- 压缩关键路径的工期，是在现有资源、成本和任务不变的前提下，针对关键路径进行优化，结合资源、成本、时间和活动可调度性等因素调整计划。
- 准关键活动通常根据项目情况标识，例如：缓冲期小于自身周期的 10%；路径上只有一两个活动是非关键活动；有特殊依赖关系且不能完全保证前导活动准时完成的活动。

### 5.4 网络模型的遍历

- 要想清楚所有路径上活动的自由度，必须通过网络模型遍历计算活动最早开始和结束时间、最迟开始和结束时间，再通过资源因素和约束条件调整活动时间，最终形成最佳活动进度表。
- 活动节点属性可包括 ES（最早开始）、EF（最早结束）、LS（最迟开始）、LF（最迟结束）、工期、缓冲。
- 正向遍历是按照活动开始到活动结束的顺序对网络中的每个活动进行遍历，用来计算每个活动最早开始和最早结束时间。
- 最早开始时间（ES）是某项活动能够开始的最早时间；最早结束时间（EF）是某一活动能够完成的最早时间。
- 正向遍历公式：

$$EF = ES + \text{工期}$$

若一个活动有多个紧前活动，则：

$$ES_j = \max(EF_{\text{pre}})$$

- 反向遍历是按照活动结束到活动开始的倒序对网络中的每个活动进行遍历，用来计算每个活动最迟开始和最迟结束日期。
- 最迟结束时间（LF）是为了使整个项目在要求完工时间内完成，某项活动必须完成的最迟时间。
- 最迟开始时间（LS）是为了使整个项目在要求完工时间内完成，某项活动必须开始的最迟时间。
- 反向遍历公式：

$$LS = LF - \text{工期}$$

若一个活动有多个紧后活动，则：

$$LF_i = \min(LS_{\text{succ}})$$

- 关键路径上的各个活动的最早开始时间等于最迟开始时间，最早结束时间等于最迟结束时间。原因是关键活动没有任何缓冲期。
- 缓冲期（浮动时间）公式：

$$\text{缓冲期} = LF - EF = LS - ES$$

### 5.5 里程碑

- 软件是无形的产品，其开发过程的可视性比较差，控制开发过程比较困难。在制定项目进度计划时，在进度时间表上设立一些重要的时间检查点，可以在项目执行过程中利用这些检查点对项目进程进行检查和控制。这些重要的时间检查点称作项目的里程碑。
- 里程碑一般是项目中完成阶段性工作的标志，标志着上一个阶段结束、下一个阶段开始，将一个过程性的任务用一个结论性的标志来描述，明确任务的起止点。
- 里程碑定义了当前阶段完成的标准（Exit Criteria）和下个新阶段启动的条件或前提（Entry Criteria）。
- 里程碑特征：里程碑有层次性；不同类型项目的里程碑可能不同；不同规模项目的里程碑数量不一样，里程碑可以合并或分解。
- 里程碑是一个以目标为导向的关键检查点。当一系列任务或活动完成，经过质量评审并得到认可后，才可以说到达了一个里程碑的完成。
- 项目跟踪是指在规定的时间间隔内对项目进行检查，比较进度与估算计划之间的差异，并根据差异进行调整。原则上检查间隔永远不要超出可控范围。
- 建立里程碑的方法：
  1. 设立合理的里程碑检查点。小型项目可把阶段完成点设为里程碑；大型项目应将长阶段分解成多个子里程碑；子里程碑之间间隔时间以不超过 2 周为宜；关键路径上一定要设立里程碑。
  2. 制定里程碑的完成目标。每个里程碑应有明确目标或交付物。
  3. 明确里程碑的验证标准。里程碑用于确认项目完成进度，需要清晰标准验证是否达到。
  4. 确认里程碑的利益相关人。应清楚定义负责人和相关人员的责权范围。
  5. 标识里程碑的进度百分比。预估每个里程碑占项目总进度的百分比，并随项目动态变化调整。
- 在敏捷 Scrum 中，每个用户故事都要有 AC（Acceptance Criteria）验收标准，每个用户故事完成、迭代完成和项目发布都有 DoD（Definition of Done）完成标准。
- 燃尽图（Burn Down Chart）是在项目或迭代完成之前，对需要完成的工作量的一种可视化表示。燃尽图有 Y 轴（剩余工作量）和 X 轴（时间），理想情况下随着剩余工作的完成烧尽至零。
- 管理里程碑应注意：重点关注、提前定期检查、及时总结。
- 对大型或复杂软件项目，需要在总项目里程碑框架下设置子项目里程碑加强管理。若当前里程碑延迟过长，应重新评估并重新设置后续里程碑日期，以确保产品质量。

### 5.6 进度计划编制

- 无论应用传统瀑布模式还是敏捷开发，进度计划都是进度控制和管理的依据。瀑布模式按整个项目计划，敏捷模式按迭代计划。
- 进度表的编制与确定，应根据项目网络图、估算的活动工期、资源需求、资源共享情况、活动最早和最晚时间、风险评估计划、活动约束条件等统一考虑。
- 制定软件项目进度计划一般分两个阶段：
  1. 在软件产品需求范围确定之前的初步进度时间表。
  2. 在软件产品需求范围确定之后的详细进度时间表。
- 进度计划的编制和更新是一个由粗到细的求精过程。
- 进度计划编制过程：先建立主进度表，划分主要阶段和主要里程碑；再细化各阶段活动或任务、活动资源、活动成本、活动排序、活动历时、活动关系和约束条件，形成详细进度表；最后根据资源分配、成本估算等约束整合为完整详细的进度计划表。
- 倒排进度是在最终发布日期已确定且不能更改情况下制定计划的一种方法。若必须以时间限制为前提，必须充分考虑风险并提前做好准备。无论正推还是倒排，都要先确定主要活动和关键检查点，再在里程碑之间正排细化。
- 进度计划编制结果应包括：项目具体活动及其相互依赖关系；每项活动的计划开始日期和期望完成日期；活动负责人；资源安排；备用进度计划；进度风险估计和规避计划。
- 进度编制策略：
  1. 重视与客户的沟通。
  2. 进度计划最好按需制定，不应简单以时钟驱动。
  3. 项目组成员共同参与制定项目进度计划。
  4. 任务分解与并行化，尽力挖掘可并行开展的任务。
  5. 任务、人力资源、时间分配要与进度相协调，避免“宇宙大爆炸式集成”，应做到持续集成。
  6. 项目工作安排一定要责任到人。
  7. 工作量分布要合理：需求分析约占 10%~25%；软件设计约占 20%~25%；软件编码约占 15%~20%；测试及缺陷修正可高达 30%~40%。
  8. 充分利用历史数据。
  9. 考虑相关风险，计划意外事故缓冲时间，根据经验增加 10%~15% 的缓冲时间较合理。
  10. 制定和使用进度计划检查清单。
- 常用进度计划方法：关键路径法（CPM）、计划评审技术（PERT）、甘特图法、表格表示法。
- PERT 理论基础是假设项目持续时间以及整个项目完成时间是随机的，且服从某种概率分布，通常假定为 β 分布。PERT 可以估计整个项目在某个时间内完成的概率。
- PERT 三点估计：
  - 乐观时间 a：任何事情都顺利时完成工作的时间。
  - 最可能时间 m：正常情况下完成工作的时间。
  - 悲观时间 b：最不利情况下完成工作的时间。
- PERT 期望值与方差：

$$\mu_i = \frac{a_i + 4m_i + b_i}{6}$$

$$\sigma_i^2 = \left(\frac{b_i - a_i}{6}\right)^2$$

- 项目按规定日期完成的概率：

$$z = \frac{D - \mu_M}{\sigma_M}$$

其中，$D$ 为目标完工日期，$\mu_M$ 为关键线路上各项工作期望值之和，$\sigma_M$ 为关键线路标准差：

$$\sigma_M = \sqrt{\sum \sigma_i^2}$$

- CPM 被称为肯定型网络计划技术，以经验数据为基础确定各项工作时间，并以缩短时间、提高投资效益为目的。
- PERT 被称为非肯定型网络计划技术，把各项工作的时间作为随机变量处理。
- 甘特图是最常见的进度表示法，用水平线段在日历时间轴上描述任务起止时间和进度安排。优点是简单、明了、直观，易于跟踪和管理。
- 表格表示法比较适用于小型项目；大型项目活动多，用表格不够直观，不便管理。
- 软件项目进度计划审查步骤：进度计划的单元模块评审；进度计划的完整评审；修改项目进度计划；批准项目进度计划。
- 项目执行过程中的进度表更改应经过变更控制流程，不能随意修改或变动。
- 经过批准的进度计划就是变更控制的基线（Baseline）。基线为项目实施提供控制、追踪项目进度的依据。

### 5.7 进度和成本控制

- 进度和成本控制的基础还是计划。项目预算提供的成本基线是成本控制的标准。将按时段估算的成本加在一起即可得出成本基准，通常以 S 曲线形式显示。
- 影响软件项目进度的因素：
  1. 从进度计划本身分析：进度计划制定不细致；约束条件和依赖环境考虑不全；工作量评估不准确。
  2. 从进度控制角度考虑：进度信息收集问题；进度监控和管理问题；计划变更调整不及时。
  3. 从团队协调方面考虑：完美主义、自尊心、想当然主义会影响进度控制。
  4. 从项目管理三角关系考虑：范围变更、成本与人力、项目质量都会影响进度。
- Scrum 每日站会严格控制在 15 分钟内，成员回答：自上次站会以来完成了什么；至下次站会前将完成什么；哪些阻碍或障碍可能影响工作。
- Brooks 法则指出：“向进度落后的项目中增加人手，只会使进度更加落后。”增加人手会因沟通成本急剧增加而降低效率，如果一定要增加人手，越早越好。
- 软件项目进度控制应贯穿整个项目生命周期。
- 项目阶段情况汇报与计划内容包括：上一阶段计划执行情况；项目问题及其跟踪；下一阶段工作计划安排；下一阶段主要风险预计和规避措施；资源申请和需协调事项。
- 进度控制手段：定期和不定期的项目进度检查；制定适当的进度控制流程；调整各种项目目标之间的平衡。
- 看板管理（Kanban）源于精益生产实践，它把工作流程形象化，把工作细分，写在卡片上，贴在状态墙上，显示任务在工作流程中的状况。
- 看板可视化状态墙使所有团队成员实时了解项目计划和进展情况。
- 每个阶段都有完成标准（DoD），卡片从前一阶段拉入下一阶段必须满足该标准。
- 看板的核心机制是限制在制品（WIP）的数量。当在制品数量小于限制值时，才可以从前一阶段拉入新的工作。
- 限制 WIP 的好处：加速价值流动；暴露问题。
- 累积流量图（CFD）用于度量价值流动效率：X 轴是时间，Y 轴是需求或任务累积数量；两条曲线的垂直距离表示某时刻 WIP 数量；水平间距表示任务周期时间；曲线斜率表示价值交付速率或吞吐量。
- 影响软件项目成本的因素：项目质量、项目管理水平、人力资源。
- 质量不足或质量过剩都会造成成本增加。质量成本包括故障成本和预防/鉴定成本，质量成本管理目标是找到最佳平衡点，使总质量成本最低。
- 故障成本是弥补软件质量缺陷发生的费用；预防/鉴定成本是保证和提高质量而消耗的费用。
- 挣值管理（EVM）是项目管理领域中非常有效的成本控制工具。只报告时间进度不够，挣值管理可以测量项目进度和成本绩效。
- 挣值管理基本参数：

$$PV = BAC \times \text{计划进度完成百分比}$$

$$EV = BAC \times \text{实际进度完成百分比}$$

其中，BAC 为完工预算，PV 为计划值，EV 为挣值，AC 为实际成本。

- 挣值管理评价指标：

$$SV = EV - PV$$

SV > 0 表明进度超前，SV < 0 表明进度滞后。

$$CV = EV - AC$$

CV > 0 表明成本节约，CV < 0 表明成本超支。

$$SPI = \frac{EV}{PV}$$

SPI > 1 表明进度超前，SPI < 1 表明进度滞后。

$$CPI = \frac{EV}{AC}$$

CPI > 1 表明成本节约，CPI < 1 表明成本超支。

- 挣值法的前提是计划和实际进度百分比要估算比较准确。
- 进度控制目标与成本控制目标、范围控制目标、质量控制目标是对立统一的关系。进度和成本控制都按计划来控制项目变化，并且必须结合范围控制、质量控制、风险控制等过程，保证各个过程控制相协调。

## 二、本章总结

本章的主线是“先计划，再控制”。项目要能按期、按预算完成，首先要把范围转化为可管理、可检查的活动；再识别活动依赖关系，画出网络图；然后用关键路径、正向/反向遍历、缓冲期等方法确定项目总工期和各活动自由度；在此基础上设置里程碑、编制进度计划，并经过评审批准形成基线；最后在执行中持续跟踪进度与成本偏差，必要时通过变更控制调整计划。

进度管理的技术重点包括活动标识、活动排序、AON/AOA 网络图、关键路径法、网络遍历、里程碑、PERT、甘特图、看板。成本管理的技术重点是成本基线、质量成本平衡和挣值管理。两者不能孤立处理，因为进度、成本、范围、质量相互制约，项目管理的核心就是在这些目标之间进行平衡。

- 计算高频：根据活动表画 AON；计算每条路径工期；找关键路径；用正向遍历计算 ES/EF；用反向遍历计算 LS/LF；计算缓冲期；用 PERT 三点估计计算期望、方差、完工概率；用 EVM 计算 PV、EV、SV、CV、SPI、CPI 并判断进度/成本状态。
- 判断易考点：
  - 关键路径是工期最长的路径，关键活动缓冲期为 0。
  - 缩短非关键路径活动不一定能缩短项目总工期；要压缩总工期，应优先优化关键路径。
  - AON 用节点表示活动；AOA 用箭线表示活动，且只能表示 FS 关系。
  - 虚拟活动不消耗时间和资源。
  - 一个活动有多个紧前活动时，ES 取各紧前活动 EF 的最大值；一个活动有多个紧后活动时，LF 取各紧后活动 LS 的最小值。
  - 里程碑不是普通任务，而是阶段性结果的关键检查点，必须有验收标准。
  - 经过批准的进度计划是变更控制基线，不能随意修改。
  - WIP 限制能加速价值流动并暴露问题。
  - SV/SPI 判断进度，CV/CPI 判断成本；SPI、CPI 大于 1 通常是有利状态。

## 四、习题及答案+解析> [!TIP] 答题提示
> 进度成本类题目优先画清活动关系，再算关键路径、时差、偏差和控制措施。



1. **如何正确标识出软件项目活动？**

   答：正确标识软件项目活动应从项目范围和WBS开始，把可交付成果逐步分解为可执行、可估算、可分派和可检查的活动。活动不宜过大，否则难以跟踪；也不宜过细，否则管理成本过高。一般应先明确项目阶段，如需求、设计、开发、测试、部署、验收；再识别每个阶段的具体活动，如需求访谈、原型评审、数据库设计、接口开发、单元测试、用户培训等。标识活动时还要考虑依赖关系、资源约束、质量活动和管理活动，不能只列编码任务。以学校教学管理系统为例，除学生管理模块开发外，还应列出数据字典确认、角色权限设计、历史数据导入、测试用例编写和验收演示等活动。

2. **各个软件项目活动之间有哪些几种依赖关系？请结合你身边的项目举例说明。**

   答：项目活动之间常见依赖关系包括完成-开始、开始-开始、完成-完成和开始-完成，其中最常用的是完成-开始，即前一活动完成后后一活动才能开始。依赖来源可以分为强制依赖、选择性依赖和外部依赖。强制依赖由客观逻辑决定，例如数据库表结构未确定前，数据导入程序难以完成；选择性依赖来自团队经验，例如先做原型评审再开发界面；外部依赖来自项目外部，例如服务器采购完成后才能部署试运行。以学校教学管理系统为例，需求访谈完成后才能形成需求规格说明；课程管理模块可以与学生管理模块并行开发；系统测试必须等主要模块集成后进行；用户培训通常依赖试运行环境部署完成。

3. **请依据下表的活动历时和活动关系画出前导网络图和箭线图，并指出关键路径及其各个活动的缓冲时间。**

4. **什么是里程碑？如何设定里程碑？里程碑的验收标准为什么重要？结合你生活或者工作设立一个小项目（例如旅游计划和实施），试试如何设立里程碑及验收标准。**

   答：里程碑是项目中的重要节点，通常代表某个阶段性成果完成或关键决策点到达。里程碑本身一般不表示持续时间，而表示“是否达到某个状态”。设定里程碑时应选择对项目控制有意义的节点，如需求确认、原型评审通过、核心功能完成、系统测试通过、用户验收完成等。验收标准重要，是因为没有清晰标准，里程碑就只是日期，无法判断成果是否真正完成。以学校教学管理系统为例，可设里程碑：第2周完成需求确认，标准是需求规格说明经教务处签字；第4周完成原型评审，标准是主要用户确认页面流程；第10周完成核心功能开发，标准是学生、课程、选课、成绩模块通过集成测试；第16周完成验收，标准是验收测试通过且遗留缺陷均有处理计划。

5. **影响软件进度和成本的因素有哪些？哪些是你完成的项目中遇到过的？请结合你的项目进行分析、找到切实可行的解决办法。**

   答：影响软件进度和成本的因素包括需求变化、估算偏差、人员能力与稳定性、技术难度、沟通效率、质量问题、外部依赖、工具环境和管理成熟度。学校教学管理系统中最容易遇到的是需求变化和数据质量问题。例如教务处在开发中期提出新增按年级、班级、课程维度的复杂统计报表，会增加设计、开发和测试工作；历史学生数据格式不统一，也会拖慢数据导入。解决办法包括建立变更控制流程，要求新增需求说明影响范围、优先级和是否替换原有功能；采用迭代交付，先保证核心流程上线，再扩展报表；提前进行数据摸底和清洗；设置风险储备；通过每日站会和看板及时暴露阻塞，避免问题积累到后期。

6. **总结一下看板管理的特点和好处？**

   答：看板管理的特点是把工作项可视化，通过“待办、进行中、测试中、已完成”等列展示任务状态，并限制在制品数量，推动团队按流动效率完成工作。它强调拉动式工作，即成员在有能力处理时从上游拉取任务，而不是被动堆积任务。看板的好处包括：一是透明，所有人都能看到当前任务、负责人和阻塞；二是便于发现瓶颈，例如测试列长期堆积说明测试资源不足；三是能减少多任务切换，通过限制进行中任务提高完成率；四是适合持续交付和运维类工作。学校教学管理系统开发中，可把需求、开发、代码评审、测试、待发布、已完成可视化，项目经理据此跟踪进度并协调资源。

7. **什么是挣值管理？通过一个具体项目来应用这种方法。**

   答：挣值管理是一种综合范围、进度和成本的项目控制方法，核心指标包括计划价值PV、挣值EV和实际成本AC。PV表示按计划到某时点应完成工作的预算价值，EV表示实际已完成工作的预算价值，AC表示实际花费。由此可计算进度偏差 `SV = EV - PV`、成本偏差 `CV = EV - AC`、进度绩效指数 `SPI = EV / PV`、成本绩效指数 `CPI = EV / AC`。例如学校教学管理系统预算为30万元，计划第8周完成50%工作，则PV为15万元；实际完成40%，EV为12万元；实际花费14万元，AC为14万元。则SV=-3万元，说明进度落后；CV=-2万元，说明成本超支；SPI=0.8，CPI=0.86。项目经理应分析落后原因，调整范围、资源或迭代计划。
