## 2. 按教材顺序总结概念

### 2.1 项目建议书

- 定义：项目建议书是项目启动前用于说明项目必要性、目标、范围、初步方案、资源需求、风险和预期收益的文件。
- 作用：为立项审批和投资决策提供依据。
- 答题框架：为什么做、做什么、大致怎么做、需要什么资源、有什么收益和风险。

### 2.2 项目可行性分析

**可行性分析的目的**

- 判断项目能否实施、是否值得实施、风险是否可接受。
- 为立项、投标、合同和资源安排提供依据。

**常见分析维度**

| 维度 | 关注问题 |
| --- | --- |
| 技术可行性 | 技术路线、平台、工具、团队能力能否支撑项目 |
| 经济可行性 | 投入成本与预期收益是否合理 |
| 操作可行性 | 用户是否愿意使用，流程是否能落地 |
| 进度可行性 | 交付期限是否现实 |
| 法律合规可行性 | 数据、版权、合同、安全和监管要求是否满足 |
| 风险可行性 | 关键风险是否能识别、评估和控制 |

**可行性结论**

- 建议实施。
- 修改方案后实施。
- 暂缓实施。
- 不建议实施。

### 2.3 项目投标

- 投标内容：技术方案、商务报价、项目计划、人员安排、质量保障、风险应对和服务承诺。
- 重点理解：投标不是单纯报价，而是证明团队能在约束下完成交付。
- 易错点：低价中标但范围、质量和进度不可行，会给后续履约带来风险。

### 2.4 软件项目合同条款评审

**合同评审重点**

- 范围和交付物是否明确。
- 验收标准是否可操作。
- 费用结算和付款节点是否清楚。
- 变更处理机制是否写明。
- 知识产权、保密、违约责任和维护服务是否明确。

**考试提示**

- 合同评审的目的不是“挑毛病”，而是避免合同承诺与实际能力、范围和风险不匹配。

### 2.5 软件开发模型

| 模型/方法 | 特点 | 适用场景 |
| --- | --- | --- |
| 瀑布模型 | 阶段顺序清晰，文档完整 | 需求稳定、合规要求强 |
| 快速原型模型 | 先用原型澄清需求 | 用户需求不清 |
| 增量模型 | 分批交付可用功能 | 需要尽早产生价值 |
| 迭代模型 | 多轮开发逐步完善 | 需求逐步明确 |
| XP | 测试驱动、结对编程、持续集成、重构 | 需要强化工程质量 |
| BDD | 用行为场景描述需求和验收标准 | 业务沟通要求高 |
| FDD | 围绕功能列表计划、设计和构建 | 功能划分清楚 |
| Scrum | Sprint、站会、评审、回顾、产品待办列表 | 需要敏捷项目组织 |

**对比记忆**

- Scrum 偏“过程和组织管理”。
- XP 偏“工程实践和代码质量”。
- BDD 偏“行为和验收场景”。
- FDD 偏“功能列表和功能交付”。

### 2.6 软件项目组织结构和人员角色

| 组织结构 | 优点 | 缺点 |
| --- | --- | --- |
| 职能型 | 专业管理强，人员归属稳定 | 跨部门协调慢，项目经理权力弱 |
| 项目型 | 响应快，项目经理权力强 | 资源可能重复，项目结束后人员安置压力大 |
| 矩阵型 | 兼顾专业部门和项目协作 | 双重汇报，容易产生资源冲突 |

**软件项目常见角色**

- 项目经理：整合目标、资源、进度、成本、质量和风险。
- 需求/业务分析人员：理解业务并转化为需求。
- 架构师/开发人员：设计并实现系统。
- 测试人员：验证功能、质量和缺陷修复。
- QA：关注过程是否符合标准。
- QC：关注产品检查和缺陷控制。
- 用户代表：确认需求和验收标准。

### 2.7 软件项目干系人

- 定义：会影响项目或受项目影响的个人、团队或组织。
- 常见对象：客户、最终用户、项目发起人、管理层、开发测试团队、运维、供应商、监管方。
- 管理步骤：识别干系人，分析权力/利益/态度/影响，制定沟通和参与策略。

### 2.8 软件项目启动动员会

- 作用：宣布项目正式启动，统一目标、范围、角色、计划、沟通机制和团队纪律。
- 输出：会议纪要、角色责任、近期任务、沟通规则、风险和问题清单。
- 易考点：启动会不是形式会议，而是把项目从“准备”推入“执行”的组织动作。

<a id="chapter-summary"></a>
## 3. 本章总结

本章讲的是项目正式开发前的准备工作。软件项目不能从编码开始，而要先判断项目是否有价值、是否可行、合同是否清楚、开发模型是否合适、组织结构和角色是否明确、干系人是否识别到位。启动阶段做得越扎实，后续需求变更、进度延误、合同争议和沟通冲突的概率越低。

<a id="key-points"></a>
## 4. 本章重点内容

- 项目建议书回答立项理由、目标、范围、方案、资源和收益。
- 可行性分析重点：技术、经济、操作、进度、法律、风险。
- 合同评审重点：范围、验收、费用、变更、知识产权、违约责任。
- 开发模型选择要看需求稳定性、风险、交付节奏和用户参与程度。
- XP 偏工程实践，Scrum 偏敏捷管理框架，BDD 偏行为验收表达，FDD 偏功能列表驱动。
- 组织结构：职能型、项目型、矩阵型；软件项目常用矩阵型或跨职能团队。
- QA 是过程质量保证，QC 是产品质量控制。
- 干系人管理从识别、分析、沟通和期望管理开始。

<a id="exercises"></a>
## 5. 习题及答案+解析

1. 软件项目的可行性分析包括哪些方面？影响决策的关键因素又是什么？

   答案：包括技术、经济、操作、进度、法律合规和风险可行性。关键因素包括业务价值是否明确、投资回报是否合理、核心需求是否稳定、技术难点是否可突破、组织是否有资源支持、主要干系人是否达成一致。若项目价值不清或关键风险不可控，应谨慎启动。

   解析：可行性分析不是只算钱，还要判断能不能做、值不值得做、能不能落地。答题时按维度展开最稳。

2. 项目的组织结构主要分哪几类？软件项目的组织结构通常采用哪一类？

   答案：常见组织结构包括职能型、项目型和矩阵型。职能型按部门划分，专业管理强但协调慢；项目型围绕项目组建团队，项目经理权力大但资源可能重复；矩阵型介于两者之间，成员既属于职能部门也参与项目。软件项目通常采用矩阵型或轻量项目型/跨职能团队。

   解析：比较题要写优缺点。软件开发需要多角色协作，所以矩阵型和跨职能团队更常见。

3. 软件开发模型有哪些？各自的特点是什么？针对你的项目会选择哪个模型？

   答案：常见模型包括瀑布、快速原型、增量、迭代、螺旋和敏捷。瀑布阶段清晰但对变化响应慢；原型适合澄清需求；增量分批交付；迭代逐步完善；螺旋强调风险；敏捷强调用户参与、短周期迭代和持续反馈。学校教学管理系统可选增量迭代结合 Scrum，先交付登录、学生和课程管理，再逐步交付选课、成绩和报表。

   解析：模型选择没有绝对答案，要说明项目特点和选择理由。需求不完全稳定时，增量迭代比纯瀑布更合适。

4. 针对 XP 和 Scrum 两种开发模式进行比较，了解它们有什么不同和各自的优势。

   答案：Scrum 偏项目组织和过程管理，强调产品负责人、Scrum Master、开发团队、产品待办列表、Sprint 计划、每日站会、评审会和回顾会。XP 偏工程实践，强调结对编程、测试驱动、持续集成、简单设计、重构、小版本发布和代码集体所有。Scrum 有利于管理迭代和协作，XP 有利于提升代码质量和响应变化。

   解析：一句话记忆：Scrum 管“怎么组织迭代”，XP 管“怎么把代码做好”。

5. 针对 BDD 和 FDD 两种开发模式进行比较，了解它们有什么不同和各自的优势。

   答案：BDD 强调用业务人员、测试人员和开发人员都能理解的行为描述表达需求，常用“给定-当-那么”形式，优势是需求沟通和验收清晰。FDD 强调围绕功能列表进行建模、计划、设计和构建，优势是功能导向明确、便于进度跟踪和责任划分。BDD 更关注用户行为和验收场景，FDD 更关注功能清单和实现流程。

   解析：BDD 的关键词是“行为”和“验收场景”，FDD 的关键词是“功能”和“功能列表”。

6. 用一句话概括软件项目经理的作用是什么，软件项目经理应具备哪些能力和素质？

   答案：软件项目经理是在既定目标和约束下整合人员、技术、进度、成本、质量和风险，使项目成果满足客户和组织期望的人。应具备计划、组织协调、沟通、风险管理、质量意识、成本意识、决策能力，并具备责任心、诚信、抗压能力、学习能力和团队领导力。

   解析：项目经理不是“催进度的人”，而是整合资源、平衡冲突、排除障碍和推动目标达成的人。

7. 什么是软件项目相关利益人？如何对他们进行分析和管理？

   答案：软件项目相关利益人即干系人，是会影响项目或受项目影响的个人、团队或组织，如客户、最终用户、发起人、项目经理、开发测试团队、运维、供应商、管理层和监管方。管理时先识别干系人，再分析角色、利益诉求、权力和影响，最后制定沟通和参与策略，如定期汇报、需求评审、原型演示和用户培训。

   解析：干系人管理的答题结构是“识别-分析-沟通-管理期望”。不同干系人诉求不同，不能用一种沟通方式处理所有人。
