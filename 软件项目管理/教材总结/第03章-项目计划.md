## 2. 按教材顺序总结概念

### 3.1 什么是项目计划

- 定义：项目计划是对项目目标、范围、活动、资源、进度、成本、质量、风险和沟通的系统安排。
- 关键理解：计划不是一次性文档，而是项目全过程中的持续策划和控制依据。
- 软件项目特点：需求和技术不确定性强，所以计划要分层、滚动、可调整。

### 3.2 项目计划的内容

**项目计划应覆盖**

- 目标和范围：做什么、不做什么、交付什么。
- 组织和角色：谁负责、谁参与、谁决策。
- 工作分解：用 WBS 将工作拆到可估算、可分派、可检查。
- 进度和里程碑：活动顺序、持续时间、关键节点。
- 资源和成本：人员、设备、工具、预算。
- 质量和风险：质量目标、评审测试、风险识别和应对。
- 沟通和交付：沟通机制、文档、验收和移交安排。

**常见输出文档**

- 项目计划书、WBS、进度表、资源计划、成本计划、质量计划、风险计划、沟通计划。

### 3.3 项目计划的方法

**滚动计划方法**

- 做法：先做总体规划，再详细计划近期工作，远期工作保持粗粒度，随着信息增加逐步细化。
- 优点：适合需求和技术逐步明确的软件项目。
- 风险：如果缺少持续跟踪，容易变成“只看眼前，没有整体控制”。

**WBS 方法**

- 定义：工作分解结构，把项目成果和工作逐层分解为工作包。
- 好的工作包应满足：可估算、可分派、可跟踪、可验收。
- 作用：范围管理、进度计划、成本估算、责任分配和风险识别的基础。

**网络计划技术**

- 作用：分析活动依赖关系，识别关键路径和缓冲时间。
- 适用：活动依赖明显、需要控制总工期的项目。
- 易考点：网络计划不是替代 WBS，而是在 WBS 活动基础上安排顺序和时间。

### 3.4 如何有效地完成项目计划

**常见错误倾向**

- 只写计划文档，不进行后续跟踪。
- 计划过粗，无法指导执行；或计划过细，管理成本过高。
- 忽视风险、质量和沟通，只排开发任务。
- 把期望当估算，没有历史数据或专家判断支撑。
- 没有关键干系人参与，导致计划不被认可。

**有效计划原则**

- 目标明确，范围清晰。
- 分工可执行，责任可追踪。
- 估算有依据，计划留缓冲。
- 与质量、风险和变更控制结合。
- 随项目变化滚动更新。

### 3.5 计划各项内容的制定

| 计划内容 | 主要回答的问题 |
| --- | --- |
| 范围计划 | 做什么、不做什么、交付物是什么 |
| 策略制定 | 采用什么开发模型、技术路线和交付方式 |
| 资源计划 | 需要哪些人员、设备、工具和外部资源 |
| 进度计划 | 任务顺序、持续时间、里程碑和关键路径 |
| 成本计划 | 项目预算、成本构成和成本控制方式 |
| 风险计划 | 识别哪些风险，如何评估、应对和监控 |
| 质量计划 | 质量目标、标准、评审、测试和度量 |

### 3.6 项目计划工具

- 常见工具：Microsoft Project、Jira、禅道、Trello、飞书项目等。
- 工具价值：任务分派、进度跟踪、状态可视化、风险和缺陷记录、协同沟通。
- 答题提示：工具不是为了“画图”，而是让计划可执行、可检查、可调整。

## 3. 本章总结

本章的核心是“计划是项目控制的基础”。项目计划不仅是开工前的文件，更是整个项目沟通、协调和控制的依据。软件项目计划应采用滚动细化思想，用 WBS 明确范围和任务，用网络计划识别关键活动，用资源、进度、成本、质量和风险计划形成完整管理框架。

<a id="key-points"></a>
## 4. 本章重点内容

- 项目计划是持续过程，不是静态文档。
- 项目计划内容：范围、任务、组织、资源、进度、成本、质量、风险、沟通、交付。
- 滚动计划适合不确定性强的软件项目。
- WBS 是范围分解和估算基础，工作包要可估算、可分派、可检查。
- 网络计划技术用于分析活动依赖、关键路径和缓冲。
- 计划制定流程：目标范围、WBS、估算、资源、进度、成本、质量、风险、评审基线。
- 项目计划工具的价值是持续跟踪和协同，不只是画甘特图。

<a id="exercises"></a>
## 5. 习题及答案+解析

1. 为什么说项目计划不是一个文档，而是一个持续的策划过程？

   答案：项目计划会形成文档，但本质是在项目全生命周期中持续分析目标、约束、资源和风险，并根据变化不断调整。软件项目需求、技术、人员和外部环境都可能变化，早期计划只能形成总体安排；随着需求细化和开发推进，估算会更准确，风险会暴露，计划也应滚动更新。

   解析：本题关键词是“持续”“滚动”“动态控制”。答题时要强调计划用于沟通、协调和控制，而不是写完存档。

2. 深刻理解本章中所介绍的计划方法，然后讨论每种方法的优点和缺点。

   答案：滚动计划先做总体规划，再细化近期工作，优点是适应不确定性，缺点是要求持续跟踪和调整。WBS 把项目成果和工作逐层分解，优点是明确范围、责任和估算基础，缺点是分解过粗会遗漏，过细会增加管理成本。网络计划技术通过活动关系分析关键路径，优点是识别关键活动和缓冲时间，缺点是依赖较准确的活动历时估计，对频繁变化项目要结合迭代管理使用。

   解析：计划方法比较题要按“方法-优点-缺点-适用场景”答。不要只写定义。

3. 在第2章《某私立中学小型信息管理系统可行性研究报告》基础之上，制定该项目的一个总体计划，包括资源、进度、风险和质量等管理计划。

   答案：学校教学管理系统目标是在一个学期内完成学生、教师、课程、选课、成绩、考勤和统计报表等核心功能。资源计划配置项目经理1人、需求/业务分析1人、后端开发2人、前端开发1人、测试1人、兼职运维1人，校方安排教务代表和教师代表参与评审。进度可分为启动与需求2周、概要设计和原型2周、核心模块开发6周、测试与修复3周、试运行和验收3周。风险计划关注需求变更、历史数据质量、用户培训不足和上线时间冲突。质量计划包括需求评审、设计评审、代码评审、单元测试、集成测试、用户验收测试和缺陷跟踪。

   解析：总体计划题应覆盖资源、进度、风险、质量四个方面。答案不必精确到每天，但要体现阶段、角色和控制措施。

4. 熟悉一个项目计划工具，然后将上述计划输入到系统内，实现计划的信息系统管理。

   答案：可选择 Microsoft Project、禅道、Jira、Trello 或飞书项目。以 Microsoft Project 为例，先建立项目日历，设置工作日和节假日；再录入 WBS 任务，如需求调研、原型设计、数据库设计、学生管理开发、课程管理开发、成绩管理开发、系统测试、用户培训和验收上线；然后设置工期、前置关系和负责人，形成甘特图。敏捷团队也可用 Jira 建立产品待办列表，按 Sprint 安排开发和测试。

   解析：工具题不要只写“使用某工具”，要说明如何把计划要素录入系统，包括任务、工期、依赖、负责人和状态跟踪。
