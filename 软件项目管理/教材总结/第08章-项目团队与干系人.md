## 2. 按教材顺序总结概念

### 8.1 项目团队建设

| 核心内容 | 含义 | 例子 |
| --- | --- | --- |
| 制度建立与执行 | 建立共同规则 | 会议规则、代码规范、提交规范、缺陷流程 |
| 目标和分工管理 | 让成员知道目标、职责和交付标准 | 任务负责人、完成时间、验收标准 |
| 工作氛围 | 建立信任和开放沟通 | 问题及时暴露，不互相推诿 |
| 激励 | 提高成员投入和成就感 | 认可、成长机会、授权、奖励 |
| 过程管理 | 持续跟踪和改进 | 站会、看板、复盘、风险跟踪 |

**易考点**

- 团队建设不是临时动员，而是通过制度、目标、氛围、激励和过程形成持续协作能力。

### 8.2 知识传递和培训

**知识类型**

| 类型 | 特点 | 传递方式 |
| --- | --- | --- |
| 显性知识 | 可文档化、可保存 | 需求文档、接口文档、测试报告、知识库 |
| 隐性知识 | 难以完全写清，依赖经验 | 结对编程、代码评审、培训、复盘、实践 |

**培训内容**

- 业务培训：理解用户流程和业务规则。
- 技术培训：掌握框架、工具和架构。
- 流程培训：理解开发、测试、变更和发布流程。
- 质量培训：理解评审、缺陷和测试要求。

### 8.3 沟通和协作

**有效沟通原则**

- 目的清楚：为什么沟通。
- 对象明确：和谁沟通。
- 表达准确：用对方能理解的语言。
- 反馈及时：确认信息是否被理解。
- 书面确认：重要事项留下记录。

**沟通障碍**

- 信息不对称、术语差异、情绪冲突、组织层级、责任不清、反馈缺失。

**双赢沟通**

- 含义：不以压倒对方为目标，而是理解双方真实利益，在约束下寻找共同可接受方案。
- 软件项目示例：客户希望更多功能，团队担心延期，可通过优先级排序和分阶段交付实现双赢。

### 8.4 经验、知识共享

- 方式：项目复盘、技术分享、代码评审、案例库、模板、最佳实践库。
- 作用：避免重复犯错，把个人经验转化为组织能力。
- 易考点：知识共享不仅是写文档，还要让团队能复用。

### 8.5 项目绩效管理

**绩效管理循环**

```text
绩效计划 -> 执行与辅导 -> 绩效评估 -> 绩效反馈 -> 绩效改进
```

**软件团队绩效指标**

| 维度 | 例子 |
| --- | --- |
| 结果 | 用户故事完成数、里程碑达成 |
| 质量 | 缺陷数、评审问题、测试通过率 |
| 过程 | 看板更新、文档提交、流程遵守 |
| 协作 | 沟通、支持他人、承担公共任务 |
| 成长 | 学习新技术、复盘改进、知识分享 |

**易错点**

- 不能只看代码行数或加班时间。
- 绩效管理的目的应是改进和成长，而不是制造对立。

### 8.6 项目干系人管理

**干系人管理流程**

1. 识别干系人：找出影响项目或受项目影响的人和组织。
2. 分析干系人：分析权力、利益、态度和影响。
3. 制定沟通策略：决定沟通频率、内容、方式和责任人。
4. 管理期望：通过评审、演示、培训、验收等方式保持认知一致。

**常见干系人**

- 发起人、客户、最终用户、管理层、项目团队、测试团队、运维、供应商、监管方。

**答题提示**

- 高权力高利益者要重点管理。
- 最终用户要尽早参与需求、原型和验收。

<a id="chapter-summary"></a>
## 3. 本章总结

本章从“人”的角度讨论软件项目管理。软件项目不是只有任务和技术，还依赖团队协作、知识传递、沟通质量、绩效机制和干系人支持。考试中要掌握团队建设五个方面、有效沟通、双赢、绩效管理循环和干系人管理流程。

<a id="key-points"></a>
## 4. 本章重点内容

- 团队建设核心：制度、目标分工、氛围、激励、过程管理。
- 知识分显性知识和隐性知识，隐性知识需要协作和实践传递。
- 有效沟通要明确目的、对象、信息、反馈和书面确认。
- 双赢不是各自完全满意，而是在约束下实现共同利益最大化。
- 绩效管理循环：计划、执行辅导、评估、反馈、改进。
- 软件团队绩效应看价值、质量、协作和成长，不能只看代码行数。
- 干系人管理：识别、分析、沟通、管理期望。

<a id="exercises"></a>
## 5. 习题及答案+解析

1. 团队建设有几个核心内容？简单阐述一下。

   答案：核心内容包括制度建立与执行、目标和分工管理、工作氛围、激励和过程管理。制度提供协作规则；目标和分工明确责任；良好氛围提高信任；激励包括认可、成长、授权和物质奖励；过程管理通过计划、跟踪、反馈和改进保证团队持续运转。

   解析：本题按五个核心内容展开即可，注意每点说明作用。

2. 思考一下自己在哪个需求层次，如何才能激励自己？

   答案：可结合马斯洛需求层次，说明自己可能处于尊重需求和自我实现需求之间，更关注能力提升、成果认可和个人价值。激励自己的方法是把大目标拆成小目标，如完成需求分析、原型、登录模块和测试报告；主动承担有挑战任务，通过反馈、复盘和同伴监督保持动力。

   解析：开放题要把理论和个人情境结合，不能只背马斯洛层次。

3. 在软件开发过程中，信息和知识是如何传递的？

   答案：通过正式和非正式渠道传递。正式渠道包括需求规格、设计文档、接口文档、会议纪要、任务看板、缺陷单、测试报告、代码注释和知识库；非正式渠道包括面对面沟通、即时消息、结对编程、代码评审讨论、站会和经验分享。显性知识适合文档化，隐性知识需要协作和实践传递。

   解析：答题要区分正式/非正式、显性/隐性，体现软件项目知识传递的复杂性。

4. 请结合身边的事情举例说明沟通管理的重要性。

   答案：以学校教学管理系统为例，教务处可能认为“成绩管理”包括录入、修改、审核、排名和导出，而开发团队可能只理解为录入和查询。如果前期没有充分沟通，开发完成后会出现返工。有效沟通应包括需求访谈、原型演示、评审确认、会议纪要和变更记录。

   解析：沟通管理的价值是让目标、范围、责任和期望一致，并让决策可追踪。

5. 文中总结了有效沟通的几个好习惯，请结合实际生活中的案例再总结出几点。

   答案：好习惯包括提前明确目的、使用对方能理解的语言、重要事项书面确认、倾听后回应、及时反馈、避免情绪化表达。小组作业中应把“谁做什么、何时完成、交付标准”写清，避免重复劳动或遗漏。遇到分歧先确认事实，再讨论观点。

   解析：有效沟通不是消息越多越好，而是减少误解并推动行动。

6. 双赢的概念是什么？

   答案：双赢是在合作、谈判或冲突处理中，不把对方视为必须被压倒的一方，而是理解双方真实利益，寻找双方都能获得合理收益的方案。软件项目中可通过优先级排序、分阶段交付、核心功能先上线、低优先级需求后续迭代，让客户尽早获得价值，也让团队保持可持续开发。

   解析：双赢不是无原则妥协，而是在约束下实现总体利益最大化。

7. 绩效管理是怎样的循环过程？

   答案：绩效管理包括绩效计划、绩效执行与辅导、绩效评估、绩效反馈和绩效改进。先明确目标、职责和评价标准；执行中持续观察和辅导；再依据事实和数据评估；反馈后制定改进计划，并进入下一轮循环。

   解析：绩效管理是循环，不是期末一次性打分。

8. 组织你的团队讨论一下绩效考核的有效方法并尝试去实践。

   答案：可从结果、过程、质量、协作和成长五方面考核。结果看用户故事和里程碑；过程看任务更新、会议参与、文档提交；质量看缺陷、代码评审和测试通过率；协作看沟通和公共任务；成长看学习、复盘和知识分享。可采用自评、互评和负责人评价结合，每两周轻量反馈。

   解析：软件团队绩效要多维度，防止单一指标导致刷代码行数或互相推责。

9. 干系人管理的重要性是什么？如何有效地做好干系人管理？

   答案：干系人管理重要，因为项目成功不仅取决于软件完成，还取决于关键人员是否支持、接受并使用成果。有效做法是识别所有干系人，分析其权力、利益、态度和影响，制定差异化沟通策略。高权力高利益者重点管理，最终用户通过访谈、原型和培训参与，运维人员提前交接部署和监控要求。

   解析：答题结构是“为什么重要 + 如何做”。干系人诉求不同，沟通策略也要不同。
