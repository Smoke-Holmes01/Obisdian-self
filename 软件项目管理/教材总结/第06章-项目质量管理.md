## 2. 按教材顺序总结概念

### 6.1 质量管理概述

- 定义：软件质量是软件满足明确需求和隐含期望的程度。
- 质量不只是“程序能运行”，还包括功能正确性、可靠性、易用性、性能、安全性、可维护性、兼容性和可扩展性。
- 关键理解：软件质量是在需求、设计、编码、评审、测试、部署和运维全过程中形成的，不是测试阶段单独“测出来”的。

### 6.2 项目质量的组织保证

| 角色 | 质量职责 |
| --- | --- |
| 项目经理 | 把质量目标纳入计划，协调资源和进度 |
| QA | 关注过程是否符合标准和规范 |
| QC/测试 | 发现产品缺陷，验证修复结果 |
| 开发人员 | 对设计、代码和单元测试质量负责 |
| 用户代表 | 确认需求、验收标准和使用效果 |

**易考点**

- QA 偏过程保证，QC 偏产品检查。
- 质量不是测试人员一个角色的责任，而是项目团队共同责任。

### 6.3 质量计划

**质量计划内容**

- 质量目标：功能、性能、安全、可靠性、易用性等目标。
- 质量标准：采用的规范、流程、验收标准。
- 质量活动：需求评审、设计评审、代码评审、测试、审计。
- 角色职责：谁负责评审、测试、缺陷跟踪和质量报告。
- 缺陷流程：缺陷等级、处理时限、回归测试和关闭标准。
- 度量指标：缺陷密度、测试通过率、评审问题数、返工率等。

**制定步骤**

1. 理解项目目标和客户要求。
2. 明确质量目标和质量标准。
3. 设计评审、测试和度量活动。
4. 分配质量角色和资源。
5. 建立跟踪、报告和改进机制。

### 6.4 软件评审方法和过程

**评审对象**

- 需求文档、设计文档、数据库设计、接口说明、代码、测试用例、用户手册。

**评审流程**

```text
计划评审 -> 提前准备 -> 召开会议 -> 记录问题 -> 修改跟踪 -> 确认关闭
```

**评审角色**

- 作者：解释材料并修改问题。
- 主持人：控制会议目标和节奏。
- 评审人员：发现缺陷和风险。
- 记录员：记录问题和处理结论。
- QA：监督过程和问题关闭。

**答题提示**

- 评审重点是尽早发现问题，降低后期返工成本。
- 评审不是追责会，不能把重点放在批评作者。

### 6.5 缺陷预防和跟踪分析

**缺陷预防**

- 通过培训、规范、检查表、评审、自动化测试、持续集成和过程改进减少缺陷产生。
- 思路：把质量活动前移，而不是等测试阶段集中发现问题。

**缺陷分析**

| 分析类型 | 关注问题 | 作用 |
| --- | --- | --- |
| 趋势分析 | 缺陷随时间如何变化 | 判断质量是否趋于稳定 |
| 分布分析 | 缺陷集中在哪些模块、类型或原因 | 找到质量薄弱点 |
| 根因分析 | 为什么产生缺陷 | 制定预防措施 |

**鱼骨图**

- 又称因果图，用于分析问题根本原因。
- 常见维度：人员、方法、工具、需求、技术、管理、环境。

### 6.6 质量度量

| 指标 | 含义 |
| --- | --- |
| 缺陷密度 | 单位规模中的缺陷数量 |
| 严重缺陷数 | 高影响缺陷数量 |
| 缺陷关闭率 | 已关闭缺陷占比 |
| 测试通过率 | 已通过测试用例占比 |
| 缺陷修复周期 | 从发现到关闭所用时间 |

### 6.7 过程质量管理

- 过程质量关注“软件是如何被开发出来的”。
- 常见指标：需求变更率、评审覆盖率、代码评审覆盖率、构建成功率、缺陷泄漏率、返工率、过程遵从率。
- 易考点：产品质量看结果，过程质量看过程；过程质量好通常能减少后期缺陷和返工。

<a id="chapter-summary"></a>
## 3. 本章总结

本章强调质量管理要前移和全过程化。高质量不是最后靠测试补救出来的，而是通过明确质量目标、制定质量计划、组织评审、缺陷预防、缺陷分析和过程度量持续形成。软件项目考试中常考质量计划内容、评审流程、缺陷分析、鱼骨图和过程质量指标。

<a id="key-points"></a>
## 4. 本章重点内容

- 软件质量包括功能、可靠性、易用性、性能、安全、可维护性等。
- QA 关注过程保证，QC 关注产品检查和缺陷控制。
- 质量计划包括目标、标准、活动、角色、评审测试、缺陷流程、度量指标。
- 评审流程：计划、准备、会议、记录、修改跟踪、关闭。
- 缺陷趋势分析看时间变化，缺陷分布分析看模块/类型/原因分布。
- 鱼骨图用于根因分析，可从人、方法、工具、需求、技术、管理等方面展开。
- 过程质量指标：需求变更率、评审覆盖率、构建成功率、缺陷泄漏率、返工率。

<a id="exercises"></a>
## 5. 习题及答案+解析

1. 谈谈你是如何认识软件质量的。

   答案：软件质量不仅是程序能运行，而是软件满足明确需求和隐含期望的程度。它包括功能正确性、可靠性、易用性、性能、安全性、可维护性、兼容性和可扩展性。对用户而言，质量表现为好用、稳定、数据准确；对开发团队而言，还表现为代码结构清晰、缺陷少、易维护；对组织而言，质量意味着支撑业务目标并降低长期维护成本。

   解析：答题时不要把质量等同于“没有 bug”。应从用户、开发团队和组织多个角度说明。

2. 如果某个项目是开发一个学校信息系统，如何完成该项目的质量计划？

   答案：先明确质量目标，如核心功能正确、关键页面响应小于 3 秒、重要数据零丢失、严重缺陷上线前全部关闭、用户验收通过。再确定质量标准和评审活动，包括需求评审、原型评审、数据库设计评审、接口评审和代码评审。制定测试策略，包括单元、集成、系统、性能、安全、兼容性和用户验收测试。规定缺陷管理流程，明确缺陷等级、修复时限、回归测试和关闭标准。最后安排质量角色和度量指标，如缺陷密度、测试通过率、评审问题数、返工率和用户满意度。

   解析：质量计划题要覆盖目标、标准、活动、流程、角色和指标，不能只写“加强测试”。

3. 请简要说明评审的基本流程。

   答案：评审流程包括计划、准备、评审会议、记录问题、修改跟踪和关闭。负责人确定评审对象、目的、范围、人员和时间；材料提前发给评审人员；会议中作者介绍背景，评审人员提出问题，记录员记录缺陷和改进项；会后责任人修改并说明处理结果；主持人或质量人员检查问题是否关闭，必要时复审。

   解析：评审核心是尽早发现问题，降低返工成本。考试中按流程顺序写最清楚。

4. 谈谈缺陷趋势分析和分布分析有什么不同，它们对质量管理工作有什么帮助？

   答案：缺陷趋势分析关注缺陷数量随时间或阶段变化，如新增、关闭和遗留缺陷趋势，用于判断质量是否趋于稳定、修复是否及时。缺陷分布分析关注缺陷在模块、类型、严重级别、来源阶段或原因上的分布，用于找到质量薄弱点和根因。趋势分析回答“质量是否在变好”，分布分析回答“问题主要在哪里”。

   解析：趋势看时间，分布看位置/类型/原因。两者结合才能既判断状态，又定位重点。

5. 有哪些指标可以用来测量软件过程质量？

   答案：指标包括需求变更率、需求评审缺陷数、设计评审问题关闭率、代码评审覆盖率、单元测试覆盖率、构建成功率、缺陷移除率、缺陷泄漏率、返工率、测试用例执行率、缺陷修复周期、迭代计划完成率和过程遵从率。

   解析：过程质量关注“怎么做”，不是只看最终软件。指标应能帮助发现过程是否稳定、有效和可改进。

6. 试通过鱼骨图对目前项目中的问题进行分析，找出其根本原因。

   答案：以学校教学管理系统“测试阶段缺陷多、进度延误”为例，可从人员、方法、工具、需求、技术和管理六方面分析。人员方面，开发人员不熟悉教务业务，测试介入晚；方法方面，需求评审不足，接口约定不清，单元测试少；工具方面，缺少持续集成和自动化回归；需求方面，统计口径多次变化，验收标准不清；技术方面，历史数据复杂，权限模型不足；管理方面，风险暴露不及时，任务缓冲不足。根因是需求确认和质量前移不足，改进措施包括加强原型评审、明确验收标准、建立接口规范、提前测试和数据清洗。

   解析：鱼骨图题要从多个维度找原因，最后归纳根因和改进措施。不要只停留在“开发不认真”。
