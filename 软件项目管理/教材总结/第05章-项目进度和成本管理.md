## 2. 按教材顺序总结概念

### 5.1 标识项目活动

- 来源：项目范围说明和 WBS。
- 标准：活动应可执行、可估算、可分派、可检查。
- 不应遗漏：需求、设计、评审、编码、测试、部署、培训、验收、项目管理活动。
- 易考点：软件项目活动不等于编码任务。

### 5.2 确定项目活动的次序

**活动依赖关系**

| 关系 | 含义 | 例子 |
| --- | --- | --- |
| 完成-开始 FS | 前一活动完成后，后一活动才能开始 | 需求确认后开始设计 |
| 开始-开始 SS | 前一活动开始后，后一活动才能开始 | 开发开始后测试设计也可开始 |
| 完成-完成 FF | 前一活动完成后，后一活动才能完成 | 文档编写和评审关闭 |
| 开始-完成 SF | 前一活动开始后，后一活动才能完成 | 较少使用 |

**依赖来源**

- 强制依赖：由逻辑或技术决定。
- 选择性依赖：由经验或管理策略决定。
- 外部依赖：由项目外部事项决定，如采购、第三方接口。

### 5.3 关键路径分析

**核心概念**

- 关键路径：网络计划中持续时间最长、决定项目最短工期的路径。
- 关键活动：位于关键路径上的活动，总缓冲通常为 0。
- 缓冲时间：活动可以延误而不影响项目总工期的时间。

**常用公式**

```text
EF = ES + 活动历时
LS = LF - 活动历时
总缓冲 = LS - ES = LF - EF
```

**压缩工期方法**

- 赶工：增加资源，通常增加成本。
- 快速跟进：并行执行原本顺序执行的活动，通常增加返工风险。
- 调整范围：减少或推迟低优先级功能。
- 改进流程或工具：提高效率，但需要验证可行性。

### 5.4 网络模型的遍历

| 遍历方式 | 方向 | 计算内容 | 作用 |
| --- | --- | --- | --- |
| 正向遍历 | 从开始到结束 | ES、EF | 确定项目最早完成时间 |
| 反向遍历 | 从结束到开始 | LS、LF | 计算缓冲时间和关键路径 |

**答题步骤**

1. 先按依赖关系画网络图。
2. 正向计算最早开始和最早完成。
3. 反向计算最迟开始和最迟完成。
4. 用总缓冲为 0 判断关键路径。

### 5.5 里程碑

- 定义：项目中的重要节点，表示阶段性成果完成或关键决策点到达。
- 特点：通常不表示持续时间，而表示状态达成。
- 好的里程碑应有明确验收标准。

**示例**

| 里程碑 | 验收标准 |
| --- | --- |
| 需求确认 | 需求规格说明经客户代表确认 |
| 原型评审通过 | 主要用户确认核心页面流程 |
| 核心功能完成 | 关键模块通过集成测试 |
| 用户验收完成 | 验收测试通过并签署报告 |

### 5.6 进度计划编制

- 常用工具：甘特图、网络图、里程碑计划、迭代计划、看板。
- 进度控制方法：比较计划与实际，识别偏差，分析原因，采取纠正措施。
- 易考点：进度计划要可执行，不能只有日期，没有任务、负责人和交付标准。

### 5.7 进度和成本控制

**影响因素**

- 进度因素：需求变更、估算偏差、人员能力、技术难度、沟通效率、外部依赖。
- 成本因素：范围扩大、返工、人员效率低、质量问题、工具和环境投入、合同变化。

**看板管理**

- 特点：工作可视化、限制在制品数量、拉动式工作、发现瓶颈。
- 好处：提高透明度，减少多任务切换，帮助团队及时处理阻塞。

**挣值管理 EVM**

| 指标 | 含义 |
| --- | --- |
| PV | 计划价值，按计划应完成工作的预算价值 |
| EV | 挣值，实际已完成工作的预算价值 |
| AC | 实际成本，实际已经花费的成本 |
| SV = EV - PV | 进度偏差，负数表示进度落后 |
| CV = EV - AC | 成本偏差，负数表示成本超支 |
| SPI = EV / PV | 进度绩效指数，小于 1 表示进度效率低 |
| CPI = EV / AC | 成本绩效指数，小于 1 表示成本效率低 |

**进度--成本平衡**

- 缩短工期通常增加成本或质量风险。
- 控制成本可能需要压缩范围或延长工期。
- 项目经理要在范围、进度、成本和质量之间做取舍。

<a id="chapter-summary"></a>
## 3. 本章总结

本章重点是用活动网络和数据指标控制项目进度与成本。进度管理从活动识别、排序和关键路径分析开始，通过里程碑、甘特图、看板等方式跟踪执行。成本管理不能脱离进度和范围，挣值管理提供了同时判断进度偏差和成本偏差的方法。软件项目管理者要避免只看日期，还要看完成的实际价值和花费。

<a id="key-points"></a>
## 4. 本章重点内容

- 活动识别基于 WBS，活动应可估算、可分派、可检查。
- 活动依赖关系：FS、SS、FF、SF，最常用 FS。
- 关键路径是最长路径，决定项目总工期；关键活动总缓冲为 0。
- 正向遍历算 ES/EF，反向遍历算 LS/LF，总缓冲 = LS - ES = LF - EF。
- 里程碑要有明确验收标准，不能只是日期。
- 看板特点：可视化、限制 WIP、发现瓶颈、拉动式流动。
- 挣值管理：`SV = EV - PV`，`CV = EV - AC`，`SPI = EV / PV`，`CPI = EV / AC`。

<a id="exercises"></a>
## 5. 习题及答案+解析

1. 如何正确标识出软件项目活动？

   答案：应从项目范围和 WBS 开始，把可交付成果分解为可执行、可估算、可分派和可检查的活动。先明确需求、设计、开发、测试、部署、验收等阶段，再识别具体活动，如需求访谈、原型评审、数据库设计、接口开发、单元测试、用户培训等。还要考虑依赖关系、资源约束、质量活动和管理活动，不能只列编码任务。

   解析：活动标识题要强调“从 WBS 来”和“不能遗漏非编码活动”。软件项目中评审、测试、部署、培训同样是活动。

2. 各个软件项目活动之间有哪些依赖关系？请结合项目举例说明。

   答案：依赖关系包括完成-开始、开始-开始、完成-完成和开始-完成。依赖来源包括强制依赖、选择性依赖和外部依赖。例如数据库表结构未确定前，数据导入程序难以完成，这是强制依赖；先做原型评审再开发界面是选择性依赖；服务器采购完成后才能部署试运行是外部依赖。

   解析：考试中常用 FS/SS/FF/SF 或中文关系表述。举例时要能说明为什么存在依赖。

3. 根据活动历时和关系画出网络图，指出关键路径及缓冲时间。

   答案：关系为：开始后执行 H1；H1 后执行 H2 和 H4；H3 需要 H1 和 H2 完成；H5 需要 H3 和 H4 完成；H6 需要 H4 完成；H7 需要 H3 和 H6 完成；H8 需要 H7 完成。关键路径为 `H1 -> H4 -> H6 -> H7 -> H8`，总工期 55 天。缓冲时间：H1=0，H2=12，H3=12，H4=0，H5=25，H6=0，H7=0，H8=0。

   解析：先正向遍历求最早时间，再反向遍历求最迟时间，最后用总缓冲判断关键活动。总缓冲为 0 的活动在关键路径上。

4. 什么是里程碑？如何设定里程碑？里程碑验收标准为什么重要？

   答案：里程碑是项目中的重要节点，代表阶段性成果完成或关键决策点到达。设定时应选择对控制有意义的节点，如需求确认、原型评审通过、核心功能完成、系统测试通过、用户验收完成。验收标准重要，因为没有标准，里程碑只是日期，无法判断成果是否真正完成。例如需求确认的标准可以是需求规格说明经教务处签字。

   解析：里程碑本身通常没有持续时间，关键是“状态”和“验收标准”。

5. 影响软件进度和成本的因素有哪些？结合项目分析解决办法。

   答案：因素包括需求变化、估算偏差、人员能力与稳定性、技术难度、沟通效率、质量问题、外部依赖、工具环境和管理成熟度。学校教学管理系统中常见问题是需求变化和历史数据质量差。解决办法包括建立变更控制流程、按优先级迭代交付、提前数据摸底和清洗、设置风险储备、通过站会和看板暴露阻塞。

   解析：影响因素要覆盖范围、人员、技术、质量和外部依赖。解决办法要对应问题，不能只写“加强管理”。

6. 总结看板管理的特点和好处。

   答案：看板通过“待办、进行中、测试中、已完成”等列展示任务状态，并限制在制品数量，推动团队按流动效率完成工作。好处包括透明、便于发现瓶颈、减少多任务切换、提高完成率，适合持续交付和运维类工作。

   解析：看板的两个核心词是“可视化”和“限制 WIP”。它关注工作流是否顺畅。

7. 什么是挣值管理？通过具体项目应用这种方法。

   答案：挣值管理综合范围、进度和成本。PV 是计划价值，EV 是挣值，AC 是实际成本。`SV = EV - PV`，`CV = EV - AC`，`SPI = EV / PV`，`CPI = EV / AC`。例如学校教学管理系统预算 30 万元，计划第 8 周完成 50%，PV=15 万元；实际完成 40%，EV=12 万元；实际花费 14 万元，AC=14 万元。则 SV=-3 万元，CV=-2 万元，SPI=0.8，CPI=0.86，说明进度落后且成本超支。

   解析：EVM 题要会判断正负：SV<0 表示进度落后，CV<0 表示成本超支，SPI/CPI 小于 1 也表示不利。
