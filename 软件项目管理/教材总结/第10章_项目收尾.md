# 第10章 项目收尾

## 一、按教材顺序保留的必备概念原话

### 1. 项目收尾概述

- 项目收尾的工作包括产品验收、移交、人员培训、付清合同款项等。
- 项目的成功验收是一系列细致工作完成到位的结果，验收的前期工作要准备充分才行。
- 对于公司内部的项目，如一些研究性项目，其项目验收过程相对简单。一旦项目完成，就可以由项目负责人组织相关的专家和领导进行项目成果的检查和验收。
- 对于敏捷的迭代项目，针对需要上线发布的迭代，要按照验收流程来走。针对不上线发布的迭代，其验收过程相对简单，基本上由项目组做演示（Demo），之后项目负责人（Product Owner）来验收用户故事的完成程度。
- 没有完成或者失败的用户故事重新进入待办列表，PO 会按优先级重新排序。下个迭代开始之前，完成当前迭代的评估。

### 2. 项目验收

#### 2.1 验收前提

- 对于项目承包商来说，只要完成了前期与客户共同认定的项目需求书中规定的各项工作，并且进行了相关的功能测试、性能测试等，同时将相关的项目文档准备完善后便可以进行项目的验收了。
- 对于客户来说，主要关心的不是合同、技术协议、需求规格说明书、用户使用手册等文档是否齐全、内容是否详尽，而是其业务是否真正地在软件系统中得以实现，并且能良好地运行，不会出现任何严重的问题。客户往往将此作为项目验收的标准。
- 针对大型的、复杂的软件项目而言，提前做一个详细的验收计划是非常必要的，可以用来指导验收工作的进行。
- 针对中小型和相对简单的软件项目而言，可能只要准备一份完整的检查清单，到时候逐项进行验收就可以了。
- 不管项目大小，项目承包方在正式验收之前，都应该做到以下几点：
  1. 完成合同要求的全部内容：即软件开发已经完成，并全部解决了已知的软件缺陷（Bug）。
  2. 完成软件系统测试：包括单元测试、集成测试、功能测试和性能测试等，并出具相关的测试报告。
  3. 各项审查全部完成：包括软件需求说明书审查、概要设计审查、详细设计审查、所有关键模块的代码审查、所有的测试脚本代码审查、对单元/集成/系统测试计划和报告的审查。
  4. 准备好相关的开发文档和产品文档。
  5. 验收测试计划准备就绪，并且已经通过评审和批准。
  6. 准备好其他验收资料，如变更记录控制文档、验收审核表等。
  7. 软件问题处理流程已经就绪。
  8. 准备好软件安装和验收测试环境。
  9. 与客户确认验收流程。
  10. 完成合同或合同附件规定的其他验收内容。
- 开发文档包括：《投标方案》《需求分析和功能要求说明书》《系统分析和技术设计书》《数据库结构和数据字典说明》《功能函数文档》《应用接口（API）说明书》等。
- 产品文档包括：《产品简介》《产品演示》《常见疑问解答》《功能介绍》《评测报告》《安装手册》《使用手册》和《维护手册》等。
- 项目承包商必须经常与客户积极主动地沟通。越是在项目的后期及验收阶段，这种沟通需要越加频繁，再次明确验收前需要完成的工作，及时处理沟通中出现的问题，还要与客户商定一个相对固定的验收期限，使双方都继续朝着这个方向去努力，防止无限期的拖延。

#### 2.2 验收测试

- 验收测试（Acceptance Testing）或用户验收测试（User Acceptance Testing, UAT）是软件开发结束后，相关的用户和/或独立测试人员根据验收测试计划，在软件产品投入实际应用以前进行的最后一次质量检验活动。
- 验收测试的目的是检验软件产品是否符合预期的各项要求，以及用户能否接受，即软件的功能和性能是否如同用户所合理期待的那样。
- 验收测试不只是检验软件某个方面的质量，而是要进行全面的质量检验，并且要决定软件是否合格，因此验收测试是一项严格的正式测试活动。
- 验收测试需要根据事先制订的验收计划，进行功能检测、质量鉴定和资料评审等活动。
- 在敏捷项目中，项目验收内容应该都包含在发布定义（Release Definition of Done）中了。项目验收的这些内容都是在项目计划时定义好的，如果有进一步要求，应该及时通过变更管理，并在项目验收标准中更新。

#### 2.3 功能检测

- 客户依据项目合同内容、验收标准和相关的需求功能说明书，对所要求达到的成果进行验证，确保功能和接口与需求说明的一致性。
- 范围检测是项目验收中对于该项目实际功能的一个全局性的检查，在整个项目验收中，它将对该项目每一个功能模块进行细致的检查，找出可能存在的错误、漏洞。
- 项目开发方最好能提前让客户参与一些功能性的测试。这既有利于缩短验收的时间，还有利于尽早发现一些客户关心的问题，尽早解决问题。
- 可以按照软件产品的模块或者业务内容分类划分来进行阶段性验收，当完成几个相关的模块或者达到某个业务目标时，双方就这部分内容完成验收工作。
- 分阶段验收更有利于及早发现和解决问题，但是缺点是客户可能为了迎合自己的实际需要，提出更多的需求变更。对于这种情况，要做好需求变更控制，确保项目的质量、时间和成本之间的平衡。

#### 2.4 质量鉴定

- 质量鉴定是依据合同中的质量条款、质量计划中的指标要求，遵循相关的质量检测标准，对项目进行质量评定。
- 质量鉴定与功能检测的侧重点不同。质量鉴定主要是对软件的性能、安全性、兼容性、系统升级和维护等方面做一个综合性的鉴定。
- 质量好的系统不仅能够满足现有的需求，而且具有可靠的安全性、系统的兼容性和良好的可扩展性等。
- 在进行性能测试和压力测试时，测试范围必须限定在那些使用频度高的和时间要求苛刻的软件功能子集中。
- 性能测试通常需要辅助工具的支持。由于开发方已经事先进行过性能测试和压力测试，因此可以直接使用开发方的辅助工具。

#### 2.5 资料评审

- 项目资料是验收的重要依据，也是项目交接、维护、后期总结和存档的凭证。
- 向客户提交的评审资料主要是产品相关说明/简介文档、测试报告、用户手册、培训文档等与客户相关的信息资料。
- 向上级提交的评审资料除了移交给客户的软件产品和资料外，还要包括软件开发管理文档，如需求说明书、概要设计说明书、项目计划、重要的会议纪要、各类检查报表以及各类重要信息记录等。
- 资料评审是最难的工作。一方面，由于变更需求等方面的压力使这项工作常常被弱化或推迟，造成持续时间变长，加大审核的难度；另一方面，资料评审中不易把握的地方非常多，每个项目都有一些特别的地方，没有一个统一的标准。

#### 2.6 验收流程

- 软件验收应是一个循序渐进的过程，要经历准备验收材料、提交申请、初审、复审，直到最后的验收合格，完成移交工作。
- 验收流程：
  1. 准备验收材料并提交验收申请：准备好验收材料后，由项目经理或项目总负责人提交给上级领导、产品经理或项目管理委员会。
  2. 初审：产品经理或市场部经理在接到验收申请后，组织公司内部专家对项目进行初审。初审的主要目的是为正式验收打基础。
  3. 成立验收委员会：初审通过后，协调并组织管理层领导、业务管理人员、客户代表、投资方代表和信息技术专家成立项目验收委员会，负责对软件项目进行正式验收。
  4. 复审（验收测试）：软件承包方/开发方以项目汇报、现场应用演示等方式汇报项目完成情况，验收委员会根据验收计划、合同内容、验收标准对项目进行评审，并形成最终验收意见。
  5. 验收合格，项目移交：验收合格之后，着手准备项目验收报告、进行项目移交和用户培训等相关收尾工作。
- 验收结果分为验收合格、需要复议和验收不合格 3 种。

#### 2.7 验收报告

- 在验收结果公布之前，项目验收委员会应该根据验收的实际情况出具验收报告。
- 验收报告内容要细致全面、客观、真实，尤其针对项目验收中的问题处理和建议，应该尽可能给出明确和详尽的信息。

### 3. 项目总结和改进

- 每个项目，不论其是否成功，都应该被当作学习的好机会，恰当地进行总结将给项目管理者、团队成员和组织带来很多收益。
- 在迭代开发中，敏捷迭代模式建议每个迭代做总结，这样更及时有效，利于持续改进。

#### 3.1 总结目的和意义

- 项目总结至少应包括以下几个目的：
  1. 分享经验：项目结束后，团队成员分享经验和体会，有助于团队建设和知识共享。
  2. 避免犯相同的错误：总结可以分析错误根源，防止重复犯错。
  3. 提出合理化建议：针对项目结果和存在的问题提出可行性建议，对将来的项目管理改进有很大帮助。
  4. 推动流程改进：通过实践发现哪些流程推进了项目运行，哪些阻碍了运行，并思考如何优化。
  5. 激励项目团队成员：有始有终，肯定团队成员做出的成绩，嘉奖优秀人员。
  6. 最佳实践的积累：通过项目的检验，将良好的工程或管理实践沉淀下来，提升公司的整体生产力。

#### 3.2 总结会议

- 项目验收完毕以后，通常项目经理会召集该项目的参与人员召开总结会。
- 大家对项目进行回顾、反思，总结、分享好的方法和实践，分析存在的问题、缺点和不足。
- 在项目管理领域通常把这个会议称作 Postmortem Meeting（事后总结会）或 Retrospective Meeting（回顾会议）。

#### 3.3 项目回顾

- 项目回顾是对所做的工作或过程作扼要的概述和评价。
- 如果项目涉及不同的子项目组，每个小组代表最好分别汇报。
- 项目总结的数据不应该等到项目结束后临时回忆，而应该在软件项目启动阶段就开始积累记录。
- 针对持续时间长的项目，阶段性总结是一个有效可行的办法。

#### 3.4 软件度量结果分析

- 在项目结束的时候，要对过程度量的结果进行适当的分析和总结。
- 在做度量结果分析时，应当拿公司内部类似的纵向项目或系列项目进行对比分析，有条件的还可以进行行业横向对比，找出差距。

常用软件项目度量指标：

| 类别 | 基本度量项 | 计算公式 / 说明 |
| :--- | :--- | :--- |
| 进度与规模度量 | 持续时间偏差（%） | （实际持续时间 - 计划持续时间）/ 计划持续时间 x 100%，持续时间不包含非工作日 |
| 进度与规模度量 | 进度偏差（%） | （实际结束时间 - 计划结束时间）/ 计划持续时间 x 100% |
| 进度与规模度量 | 工作量偏差（%） | （实际工作量 - 计划工作量）/ 计划工作量 x 100% |
| 进度与规模度量 | 规模偏差（%） | （实际规模 - 计划规模）/ 计划规模 x 100% |
| 进度与规模度量 | 软件需求稳定性指数（%） | 衡量需求在项目生命周期中的变动程度 |
| 进度与规模度量 | 发布前缺陷发现密度（个/KLOC） | 发布前所有报告的缺陷总数 / 代码规模（KLOC） |
| 进度与规模度量 | 遗留缺陷密度（个/KLOC） | 发布后发现的缺陷数 / 代码规模（KLOC） |
| 质量与成本度量 | 质量成本（%） | （评审工作量 + 返工工作量 + 缺陷修改工作量 + 测试准备工作量 + 测试执行工作量 + 培训工作量 + 质量保证工作量）/ 实际总工作量 x 100% |
| 质量与成本度量 | 返工成本指数（%） | （返工工作量 + 缺陷修改工作量）/ 实际总工作量 x 100% |
| 质量与成本度量 | 文档缺陷发现密度（个/页） | 发现的缺陷数 / 文档页数 |
| 质量与成本度量 | 测试用例缺陷发现密度（个/用例） | 发现的缺陷数 / 测试用例数 |
| 质量与成本度量 | 千行代码缺陷率（个/KLOC） | 发现缺陷数 / 千行代码行数 |
| 质量与成本度量 | 回归缺陷率（%） | 各个 Build 中回归缺陷数 / 全部缺陷数 x 100% |
| 质量与成本度量 | 无用缺陷率（%） | （重复缺陷数 + 报错/误报缺陷数）/ 全部缺陷数 x 100% |
| 代码质量度量 | 圈复杂度 | 衡量模块判定结构的复杂程度（独立线性路径条数），覆盖所有情况所需的最少用例数 |
| 代码质量度量 | 继承树深度 | 提供类在对象层次结构中继承等级的度量 |
| 代码质量度量 | 包缠绕指数 | 给出包的复杂等级；最好值为 0%（无循环依赖）；公式：2 x package_tangles / package_edges_weight x 100% |
| 代码质量度量 | 缺陷代码审查率（%） | 审查过的缺陷数 / 全部处理过的缺陷数 x 100%，建议 100% |
| 每千行代码度量 | 每千行代码文档规模（pages/KLOC） | 文档页数 / 代码千行数 |
| 每千行代码度量 | 每千行代码测试用例规模（use cases/KLOC） | 测试用例数 / 代码千行数 |
| 生产率指标 | 文档生产率（页/人天） | 文档页数 /（文档准备工作量 + 文档评审工作量 + 文档修改工作量） |
| 生产率指标 | 测试用例生产率（用例/人天） | 测试用例数 /（用例准备工作量 + 用例评审工作量 + 用例修改工作量） |
| 生产率指标 | 代码生产率（LOC/人天） | 实际代码行数 /（代码准备工作量 + 编码工作量 + 代码评审工作量 + 代码修改工作量） |
| 测试执行效率 | 自动化测试覆盖率（%） | 自动化测试用例数 / 全部测试用例数 x 100% |
| 测试执行效率 | 用例执行效率（用例/人天） | 用例数 / 测试人员工作量 |

#### 3.5 经验、体会分享

- 项目组成员多多少少都会有自己的体会和感受，这个部分就是鼓励大家把好的经验拿出来分享。
- 可以使用 Top List 的方法，如每个人总结自己在项目中做得最好的两点和最差的两点，来进行分享交流。

#### 3.6 改进和建议方案讨论

- 在总结经验教训后，根据已经取得的成绩，提出改进和建议方案。
- 在讨论后必须设定行动目标（Action Item）并指派负责人进行跟踪。
- 行动目标通常应明确行动目标、负责人（Action Owner）和截止日期（Due Date）。

#### 3.7 嘉奖和庆祝

- 项目结束了，对表现优秀的员工要给予精神上或物质上的奖励，树立榜样。
- 在经费允许的情况下举行一个庆祝会，也是团队建设和友好协作的有效途径。

#### 3.8 总结报告

- 总结报告是把总结会议的内容和讨论结果形成正式的书面文件，一方面提交给上级部门审阅，另一方面保存在项目档案中。
- 报告内容通常包括：
  1. 项目整体信息回顾、度量结果分析。
  2. 做得好的方面（Successes）。
  3. 做得差的方面（Failures / Improvements）。
  4. 改进方案和建议，包括要采取的措施和责任人。
  5. 寻求帮助信息，如工作设施或环境需要改善等。
- 写总结报告时应注意：
  1. 实事求是，成绩不夸大，缺点不缩小。
  2. 分析问题着眼点要准确，分析要深入，不隐瞒矛盾。
  3. 条理要清楚，主次分明。
  4. 剪裁得体，图文并茂，去芜存精。

### 4. 本章小结

- 收尾是项目生命周期的最后一个阶段，也是形成产品闭环的关键一步。
- 项目收尾主要分为两个过程：项目验收和项目总结。
- 项目验收重点是确保项目可以通过客户验收并顺利移交；项目总结则是实事求是地列出项目的优缺点，分析不足，找出改进和解决的方案，提高个人能力和团队的合力。

## 二、本章总结

本章围绕“项目如何结束”展开，核心逻辑是先把产品和资料交付清楚，再把项目过程中的经验教训沉淀下来。项目收尾不是简单宣布项目结束，而是包括验收、移交、培训、合同款项处理、总结和改进等一组正式活动。

项目验收关注“交付物是否被客户或组织接受”。验收前要完成合同范围、测试、审查、文档、环境、问题处理流程和验收流程确认；验收测试则从功能检测、质量鉴定和资料评审三个方面判断软件是否满足合同、需求和质量要求。验收流程通常经历提交申请、内部初审、成立验收委员会、复审或正式验收测试、出具验收报告、移交和归档。

项目总结关注“组织是否从项目中学习”。总结会议要回顾项目过程，分析度量结果，分享经验体会，讨论改进建议，设置可跟踪的行动目标，并对团队成员进行必要的嘉奖。总结报告则把这些内容形成正式文档，服务于项目档案、后续维护和组织级持续改进。

## 三、本章重点内容

1. 名词解释重点：
   - 项目收尾：项目生命周期最后阶段，包括产品验收、移交、人员培训、付清合同款项、项目总结等工作。
   - 验收测试 / UAT：软件开发结束后，用户和/或独立测试人员根据验收测试计划，在软件产品投入实际应用以前进行的最后一次质量检验活动。
   - 功能检测：依据合同、验收标准和需求功能说明书验证成果，确保功能和接口与需求说明一致。
   - 质量鉴定：依据合同质量条款、质量计划指标和质量检测标准，对性能、安全性、兼容性、升级和维护等进行综合评定。
   - 资料评审：对交付、维护、总结、存档所需项目资料进行评审。
   - Postmortem Meeting / Retrospective Meeting：项目结束后的事后总结会或回顾会议。

2. 简答题重点：
   - 项目承包方正式验收前应完成的 10 项准备。
   - 验收测试包括功能检测、质量鉴定、资料评审。
   - 软件验收流程：准备验收材料并提交申请、初审、成立验收委员会、复审（验收测试）、验收合格后移交。
   - 项目总结的 6 个目的：分享经验、避免犯相同错误、提出合理化建议、推动流程改进、激励团队成员、积累最佳实践。
   - 总结会议的主要内容：项目回顾、软件度量结果分析、经验体会分享、改进和建议方案讨论、嘉奖和庆祝。
   - 总结报告的内容和写作注意事项。

3. 判断题易考点：
   - 项目收尾不只是验收，还包括移交、培训、合同款项处理、总结和改进。
   - 客户验收更关注业务是否真正实现并能良好运行，而不只是文档是否齐全。
   - 大型复杂项目需要详细验收计划；中小型简单项目可用完整检查清单。
   - 验收测试是投入实际应用前的最后一次质量检验活动，是严格的正式测试活动。
   - 敏捷项目的验收内容应包含在发布定义（Release Definition of Done）中，新增要求应走变更管理。
   - 阶段性验收有利于及早发现和解决问题，但可能带来更多需求变更，因此必须做好变更控制。
   - 资料评审是验收的重要工作，资料是项目交接、维护、总结和存档的凭证。
   - 验收结果分为验收合格、需要复议和验收不合格 3 种。
   - 项目总结的数据应从项目启动阶段开始积累，不应到结束后临时回忆。
   - 软件度量分析应与内部类似项目纵向对比，有条件时再做行业横向对比。

4. 公式和指标易考点：
   - 持续时间偏差、进度偏差、工作量偏差、规模偏差的分子都体现“实际 - 计划”。
   - 缺陷密度类指标常用“缺陷数 / 规模”。
   - 质量成本包括评审、返工、缺陷修改、测试准备、测试执行、培训、质量保证等工作量占实际总工作量的比例。
   - 返工成本指数 =（返工工作量 + 缺陷修改工作量）/ 实际总工作量 x 100%。
   - 自动化测试覆盖率 = 自动化测试用例数 / 全部测试用例数 x 100%。

## 四、习题及答案+解析

1. **项目经理在项目收尾工作中起到什么作用？**

   答：项目经理在项目收尾中起组织、协调、确认和总结作用。首先，项目经理要确认合同或计划范围内的交付物是否完成，包括软件系统、源代码、部署文档、用户手册、测试报告和培训材料。其次，要组织验收测试和验收会议，协调客户、用户、开发、测试和运维人员处理遗留问题。再次，要推动项目文档归档、资源释放、合同结算和运维交接，保证项目成果能持续运行。最后，项目经理要组织项目总结，分析目标达成情况、进度成本偏差、质量问题、风险处理和经验教训。收尾不是简单宣布结束，而是确保项目被正式接受、责任完成、知识沉淀，并为后续运维和新项目提供基础。

2. **项目验收的前提是什么？**

   答：项目验收的前提是项目交付物已经达到约定的验收条件，并具备可检查、可运行、可交付的状态。具体包括：需求范围已经实现或经双方确认调整；关键缺陷已修复，遗留缺陷有明确处理方案；系统测试、集成测试和必要的性能、安全测试已完成；部署环境稳定，数据迁移完成或具备试运行数据；用户手册、安装部署文档、测试报告、培训材料等文档齐全；用户代表已完成必要培训；验收标准和验收用例已得到确认。学校教学管理系统中，若成绩管理、选课和报表等核心流程尚未通过测试，或教师尚不会操作系统，就不应进入正式验收，否则验收会变成问题暴露会。

3. **项目验收测试应该包括哪些方面？**

   答：项目验收测试应围绕用户业务目标和合同要求展开，主要包括功能验收、业务流程验收、数据验收、性能验收、安全验收、兼容性验收、易用性验收和文档验收。学校教学管理系统中，功能验收应覆盖登录权限、学生管理、教师管理、课程管理、选课排课、成绩录入、考勤和统计报表；业务流程验收应模拟真实教务场景，如新学期排课、学生选课、教师录入成绩、教务处导出报表；数据验收应检查历史数据导入是否完整准确；性能验收应检查多用户查询和报表生成速度；安全验收应验证不同角色权限隔离；文档验收应检查用户手册和运维手册是否可用。验收测试结果应形成正式记录，作为签署验收报告的依据。

4. **项目总结的目的有哪些？**

   答：项目总结的目的包括评估项目目标达成情况、沉淀经验教训、发现过程问题、表彰团队贡献、改进组织能力和支持后续维护。通过总结，可以比较计划与实际在范围、进度、成本、质量和风险方面的差异，分析成功原因和失败原因。项目总结还可以把隐性经验转化为组织知识，例如哪些需求沟通方法有效、哪些估算偏差较大、哪些风险应提前处理。对学校教学管理系统而言，总结可以发现需求原型评审对减少返工很有效，也可能发现数据迁移准备不足导致上线延期。项目总结不是形式化写报告，而是为了让团队和组织在下一次项目中做得更好。

5. **如何召开项目总结会议？**

   答：召开项目总结会议应提前明确目的、范围、参与人员和材料。会前，项目经理应收集项目计划、进度数据、成本数据、缺陷统计、变更记录、风险记录、验收结果和用户反馈，并让团队成员准备个人总结。会议中可按项目目标回顾、主要成果、偏差分析、问题原因、优秀实践、经验教训和改进建议的顺序进行。主持人应营造客观、开放的氛围，避免把总结会变成追责会。对问题要追溯到过程和机制，而不是停留在个人抱怨。会议结束后，应形成总结报告和行动项，明确哪些经验进入模板，哪些问题需要制度改进。学校信息系统项目还应邀请教务代表和运维人员参加，以获得用户和运维视角。

6. **软件度量的意义是什么？结合手边的项目，尝试做一下度量分析，看看有什么收获？**

   答：软件度量的意义在于用数据认识项目和产品状态，支持计划、控制、评价和改进。没有度量，项目管理容易依赖主观感觉；有了度量，团队可以发现趋势、验证问题和评估改进效果。以学校教学管理系统为例，假设项目共完成16.2 KLOC，系统测试发现81个缺陷，则缺陷密度为 `81 / 16.2 = 5 个/KLOC`；其中成绩管理模块发现25个缺陷，占比最高，说明该模块业务规则复杂，应加强评审和测试。若平均缺陷修复周期为3天，但严重缺陷为6天，说明严重缺陷需要更高优先级和资源保障。若需求变更率达到20%，说明前期需求确认不足。通过这些度量，可以得到收获：质量问题集中在复杂业务模块，需求和评审应前移，后续项目要更早制定验收标准和测试用例。
