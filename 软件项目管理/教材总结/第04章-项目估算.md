## 2. 按教材顺序总结概念

### 4.1 项目估算的挑战

- 估算困难的原因：早期需求不完整、技术方案未验证、人员能力差异大、历史数据不足。
- 关键理解：估算不是精确预测，也不是随便猜，而是基于数据、经验和模型形成有依据的判断。
- 易考点：软件估算应滚动修正，越接近交付，估算通常越准确。

### 4.2 项目估算的基本内容

| 估算类型 | 回答的问题 | 常见结果 |
| --- | --- | --- |
| 规模估算 | 要做多少 | LOC、功能点、用例点、构件数量 |
| 工作量估算 | 需要多少劳动投入 | 人月、人日 |
| 资源估算 | 需要哪些资源 | 人员角色、设备、工具 |
| 工期估算 | 多久完成 | 周期、里程碑、进度安排 |
| 成本估算 | 花多少钱 | 人力成本、软硬件、管理费、风险储备 |

### 4.3 基本估算方法

| 方法 | 核心思路 | 优点 | 局限 |
| --- | --- | --- | --- |
| 专家判断 | 依靠专家经验 | 快，适合早期 | 主观性强 |
| 类比估算 | 参考类似项目 | 简单直观 | 依赖可比项目 |
| 自下而上估算 | 逐项估算后汇总 | 细致准确 | 成本高，前提是分解充分 |
| 参数模型估算 | 用公式或模型计算 | 可重复，可解释 | 依赖模型和参数质量 |
| 三点估算 | 综合乐观、最可能、悲观值 | 能反映不确定性 | 三个值仍需合理判断 |

**三点估算公式**

```text
期望估算值 = (O + 4M + P) / 6
O = 乐观值，M = 最可能值，P = 悲观值
```

### 4.4 软件规模估算

**德尔菲法**

- 做法：多名专家匿名估算，汇总结果后反馈，再进行多轮修正，直到趋于一致。
- 优点：减少权威影响和群体压力。
- 易考点：关键词是“专家”“匿名”“多轮”“趋同”。

**代码行 LOC 估算**

- 基本公式：`工作量 = 代码规模 / 生产率`。
- 优点：直观，适合有历史代码数据的组织。
- 局限：受编程语言、编码风格、复用程度和工具影响。

**功能点分析**

- 思路：从用户可见功能出发估算规模，而不是从代码行出发。
- 优点：适合需求阶段，和编程语言无关。
- 关注对象：输入、输出、查询、内部逻辑文件、外部接口文件等。

**标准构件法**

- 思路：以可复用构件或标准模块为估算单位。
- 适用：组织已有构件库、产品线或相似模块较多的项目。

### 4.5 工作量估算

**COCOMO 方法**

- 思路：用软件规模 KLOC 和项目类型估算工作量。
- 基本有机型公式：

```text
E = 2.4 * KLOC^1.05
T = 2.5 * E^0.38
E = 工作量（人月），T = 开发周期（月）
```

**其他方法**

- 多变量模型：同时考虑规模、复杂度、人员能力、工具、环境等因素。
- 基于用例估算：从用例数量和复杂度推导工作量。
- IBM RMC：结合过程、角色和活动进行估算。
- 扑克牌估算：敏捷团队成员独立估算、讨论差异、重新出牌，形成共识。

### 4.6 资源估算

- 人员资源：项目经理、需求、设计、开发、测试、运维、QA 等。
- 非人员资源：开发环境、测试环境、服务器、工具、外包服务。
- 易错点：不能只算“开发人员”，还要算测试、管理、评审、部署和培训资源。

### 4.7 工期估算和安排

- 工期受工作量、人员投入、活动依赖、资源冲突和质量要求影响。
- 增加人员不一定线性缩短工期，因为沟通成本、培训成本和任务依赖会增加。
- 进度安排要结合里程碑、关键路径和风险缓冲。

### 4.8 成本估算

- 成本组成：人力成本、软硬件成本、培训成本、外包成本、差旅成本、管理成本、风险储备。
- 学习曲线：团队随着重复实践和经验积累，单位工作效率可能提高。
- 答题提示：成本估算不能只算工资，软件项目还有管理、质量、工具和风险成本。

<a id="chapter-summary"></a>
## 3. 本章总结

本章围绕“如何在不确定条件下做有依据的预测”。软件项目估算通常从规模开始，推导工作量、资源、工期和成本。不同方法各有适用场景，考试中要掌握 LOC、功能点、COCOMO、三点估算和扑克牌估算的基本思想。估算结果不是承诺本身，而是计划、预算、投标和风险控制的基础。

<a id="key-points"></a>
## 4. 本章重点内容

- 估算内容：规模、工作量、资源、工期、成本。
- 三点估算：`E = (O + 4M + P) / 6`。
- 德尔菲法：多轮匿名专家估算，减少权威影响和偏见。
- LOC 方法：规模/生产率 = 工作量；受语言、编码风格影响。
- 功能点：从用户功能角度估算，适合需求阶段。
- COCOMO：用 KLOC 和项目类型估算工作量。
- 扑克牌估算：敏捷团队常用，通过讨论差异形成共识。
- 成本估算不能只算工资，还要考虑工具、设备、培训、管理和风险储备。

<a id="exercises"></a>
## 5. 习题及答案+解析

1. 针对学校教学管理系统，进行功能分解。以 LOC 估算每个功能的规模。假设公司平均生产率是 450LOC/pm，平均劳动力价格是每人月 7000 美元，估算工作量及成本。

   答案：可分解为登录与权限1200 LOC、学生信息1800 LOC、教师信息1200 LOC、课程管理1500 LOC、选课与排课2500 LOC、成绩管理2200 LOC、考勤管理1400 LOC、考试管理1200 LOC、统计报表2000 LOC、系统维护与基础配置1200 LOC，总规模约 16200 LOC，即 16.2 KLOC。工作量为 `16200 / 450 = 36 人月`。人力成本为 `36 * 7000 = 252000 美元`。实际预算还应考虑服务器、工具、培训、数据迁移、管理成本和风险储备。

   解析：LOC 估算题的核心公式是“规模 / 生产率 = 工作量”，“工作量 * 人月单价 = 成本”。注意单位统一：LOC、LOC/人月、人月。

2. 使用 COCOMO 模型估算学校教学管理系统开发项目的工作量，并采用其他方法估算，然后用三点估算法导出单一估算值。

   答案：学校教学管理系统规模约 16.2 KLOC，可按基本 COCOMO 有机型估算：`E = 2.4 * KLOC^1.05`，代入得到 `E = 2.4 * 16.2^1.05 ≈ 44.7 人月`。开发周期可用 `T = 2.5 * E^0.38`，约 10.6 个月。其他方法中，LOC 生产率法得到 36 人月；类比同类系统，考虑数据迁移、报表和验收沟通可估为 52 人月。三点估算取 O=36、M=44.7、P=52，则 `(36 + 4*44.7 + 52) / 6 ≈ 44.5 人月`。对应人力成本约 `44.5 * 7000 = 311500 美元`。

   解析：本题综合 COCOMO 和三点估算。答题时要写公式、代入、结果，并说明不同方法结果可互相校验。

3. 成本和进度估算是在软件项目计划期间，也就是详细需求分析或设计之前完成的。你认为为什么会这样？是否存在不需要这样做的情况？

   答案：成本和进度估算必须在详细需求和设计前完成，因为立项、投标、合同签订和资源安排都需要先判断项目是否值得做、需要多少预算、大致何时交付。早期估算虽然不精确，但能提供决策依据，并随着项目推进滚动修正。不存在完全不需要估算的情况，只是粒度和形式不同。探索性项目可做时间盒和预算上限估算，敏捷项目可先做发布级估算，再在迭代中细化故事点和工时。

   解析：估算早于详细设计是管理决策需要。考试中要强调“早期估算不精确但必要”，并补充“滚动修正”。
