## 2. 按教材顺序总结概念

### 7.1 项目风险带来的警示

- 风险是可能影响项目目标的不确定事件或条件。
- 软件项目常见风险：需求变化、技术难题、估算偏差、人员流失、外部依赖、质量问题、进度压力、合同约束不清。
- 易考点：风险不一定已经发生；已经发生的问题通常叫“问题”或“事件”，未发生但可能发生的才是风险。

### 7.2 什么是风险管理

- 定义：风险管理是识别、分析、应对和监控风险的系统过程。
- 目标：提前看到不确定性，降低负面影响，必要时利用机会。
- 关键理解：风险管理不是消灭所有风险，而是让风险可见、可评估、可应对。

### 7.3 风险管理模型

```text
风险识别 -> 风险评估 -> 风险计划 -> 风险应对 -> 风险监控
```

| 阶段 | 主要任务 |
| --- | --- |
| 识别 | 找出可能影响目标的风险 |
| 评估 | 判断概率、影响和优先级 |
| 计划 | 制定策略、责任人和触发条件 |
| 应对 | 执行规避、减轻、转移、接受等措施 |
| 监控 | 跟踪风险状态和应对效果 |

### 7.4 风险识别

**软件风险分类**

- 需求风险：需求不清、频繁变更、用户参与不足。
- 技术风险：技术方案不成熟、性能不达标、接口不稳定。
- 人员风险：关键人员离职、能力不足、沟通不畅。
- 管理风险：计划不现实、职责不清、决策慢。
- 外部风险：供应商、政策、合同、第三方平台变化。

**识别方法**

- 检查表、头脑风暴、专家访谈、历史项目复盘、WBS 分析、假设分析、流程分析。
- 答题提示：识别风险时要鼓励透明，不应把提出风险等同于消极或推卸责任。

### 7.5 风险评估

**风险度量内容**

- 发生概率。
- 影响程度。
- 风险暴露值。
- 触发条件。
- 可控性和可检测性。

**评估方法**

| 方法 | 用途 |
| --- | --- |
| 概率-影响矩阵 | 定性排序风险优先级 |
| 期望货币值 | 量化风险的经济影响 |
| 敏感性分析 | 找出对结果影响最大的因素 |
| 决策树 | 比较不同决策路径的结果 |
| 蒙特卡罗模拟 | 通过大量随机模拟预测工期/成本分布 |

### 7.6 风险监控和规避

**风险应对策略**

| 策略 | 含义 | 示例 |
| --- | --- | --- |
| 规避 | 改变方案，避免风险发生 | 放弃不成熟技术 |
| 减轻 | 降低概率或影响 | 先做技术原型 |
| 转移 | 把风险影响转给第三方 | 购买服务或外包 |
| 接受 | 承认风险并准备应急措施 | 设置风险储备 |
| 利用 | 对机会型风险主动争取收益 | 把成熟组件用于更多模块 |

**风险监控**

- 建立风险清单。
- 定期检查风险状态、触发条件和应对效果。
- 在例会、里程碑、变更评审中更新风险。

### 7.7 风险管理的高级技术

| 技术 | 核心思想 | 易考点 |
| --- | --- | --- |
| VERT | 将风险、时间、成本放入网络模型分析 | 适合不确定路径分析 |
| 蒙特卡罗法 | 大量随机模拟得到工期/成本分布 | 输出是概率分布，不是单一值 |
| SWOT | 分析优势、劣势、机会、威胁 | 同时看内部和外部因素 |
| 关键链技术 | 在关键路径基础上考虑资源约束和缓冲 | 关注资源冲突和集中缓冲 |

### 7.8 风险管理最佳实践

- 尽早识别风险，不等问题发生。
- 风险清单要有责任人、应对策略和触发条件。
- 风险沟通要透明，避免报喜不报忧。
- 对高概率高影响风险优先处理。
- 风险管理要和计划、变更、质量、沟通结合。

<a id="chapter-summary"></a>
## 3. 本章总结

本章的核心是“主动管理不确定性”。软件项目风险往往在早期已有征兆，项目经理要通过风险识别、评估、应对和监控把风险显性化。考试中要掌握风险管理流程、风险识别方法、概率影响评估、风险应对策略，以及 VERT、蒙特卡罗、SWOT、关键链与关键路径的区别。

<a id="key-points"></a>
## 4. 本章重点内容

- 风险 = 不确定事件或条件，对目标有影响。
- 风险管理流程：识别、评估、计划、应对、监控。
- 软件项目常见风险：需求、技术、人员、进度、成本、质量、外部依赖。
- 定性评估常用概率-影响矩阵；定量评估可用蒙特卡罗、决策树等。
- 风险应对：规避、减轻、转移、接受、利用。
- VERT 结合风险、时间和成本进行网络分析。
- 关键路径看逻辑最长路径，关键链进一步考虑资源约束和缓冲。

<a id="exercises"></a>
## 5. 习题及答案+解析

1. 风险管理中，最重要的事情是什么？最难处理的问题是什么？

   答案：最重要的是提前识别和主动应对，而不是等问题发生后救火。项目经理应建立风险清单，评估概率和影响，确定责任人、应对策略和触发条件，并持续跟踪。最难处理的问题通常是人的认知和组织行为，如团队不愿暴露风险、干系人低估风险、外部风险项目组难以控制。

   解析：风险管理不是“出了问题再处理”。答题要突出提前、主动、持续和透明文化。

2. 需求变更的风险如何防范和控制？

   答案：防范上要做好需求获取和确认，通过访谈、原型、用户故事、验收标准和需求评审减少偏差；建立需求基线。控制上要走变更流程，说明变更原因、业务价值和期望时间，评估对范围、进度、成本、质量和风险的影响，再由项目负责人或变更控制委员会决策。低价值高影响变更应推迟或拒绝，必须变更的内容要同步调整计划和合同。

   解析：防范强调前期澄清和基线，控制强调流程、影响分析和决策。不要写成简单“拒绝变更”。

3. 在软件开发过程中，严格执行开发流程和规范，对防范风险有什么帮助？

   答案：流程和规范能把隐性风险显性化、前置化和可控制化。需求评审减少误解，设计评审发现架构问题，编码规范降低维护风险，代码评审发现逻辑和安全问题，版本管理避免代码丢失，测试流程降低缺陷泄漏，发布流程减少上线事故。

   解析：流程的价值是降低不确定性和提高可追溯性，不是增加形式主义文档。

4. 如何更好地应用 VERT 技术？

   答案：先建立清晰的项目活动网络，识别关键活动、可选路径和可能结果；再为活动持续时间、成本和成功概率设置合理估计；结合历史数据、专家判断和敏感性分析，找出影响最大的风险活动；最后制定应对策略，如验证原型、设置缓冲、准备替代方案。比如对历史数据迁移设置乐观、正常、失败重做等分支，计算不同路径对工期和成本的影响。

   解析：VERT 的重点是把不确定结果放进网络模型中分析，不只是普通网络计划。

5. 谈谈关键链技术和关键路径技术的异同。

   答案：两者都用于进度管理，都关注哪些活动决定项目总工期。关键路径主要基于活动逻辑关系和持续时间计算最长路径，假设资源能按计划满足；关键链在关键路径基础上考虑资源约束和人的行为因素，如多任务切换、学生综合征、帕金森定律。关键链通常把单个任务中的安全时间集中为项目缓冲、接驳缓冲和资源缓冲。关键路径适合逻辑关系明确、资源冲突少的计划分析；关键链适合资源有限和多项目共享人员环境。

   解析：关键路径看“逻辑”，关键链看“逻辑 + 资源 + 缓冲”。这是本题最重要的区分。
