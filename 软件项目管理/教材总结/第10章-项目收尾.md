## 2. 按教材顺序总结概念

### 10.1 验收

**项目收尾的含义**

- 项目收尾是项目生命周期的最后阶段。
- 目标：确认交付物被正式接受，完成合同和行政收尾，移交运维，释放资源，沉淀经验教训。
- 易考点：收尾不是简单宣布结束，而是把验收、移交、归档、结算、总结和改进做完整。

**验收定义**

- 验收是客户或用户依据合同、需求和验收标准，确认项目成果是否满足要求的过程。

**验收前提**

- 范围内交付物已经完成或经双方确认调整。
- 关键缺陷已修复，遗留缺陷有处理方案。
- 系统测试、集成测试和必要性能/安全测试已完成。
- 部署环境稳定，数据迁移完成或具备试运行数据。
- 用户手册、部署文档、测试报告、培训材料齐全。
- 用户已完成必要培训。
- 验收标准和验收用例已确认。

**验收测试内容**

| 类型 | 检查内容 |
| --- | --- |
| 功能验收 | 功能是否满足需求 |
| 业务流程验收 | 真实业务流程是否跑通 |
| 数据验收 | 数据是否完整、准确、一致 |
| 性能验收 | 响应时间、并发、稳定性 |
| 安全验收 | 权限、数据保护、越权访问 |
| 兼容性验收 | 浏览器、系统、设备适配 |
| 易用性验收 | 用户是否容易理解和操作 |
| 文档验收 | 手册、部署文档、测试报告是否齐全 |

**验收流程**

```text
提交验收申请 -> 准备验收材料 -> 执行验收测试 -> 记录问题 -> 整改复验 -> 形成结论 -> 签署验收报告
```

**验收报告内容**

- 项目基本信息、验收依据、验收范围、测试情况、问题清单、遗留问题处理、验收结论、各方签字。

### 10.2 项目总结和改进

**项目总结目的**

- 评估项目目标达成情况。
- 分析范围、进度、成本、质量和风险偏差。
- 沉淀成功经验和失败教训。
- 改进组织过程能力。
- 支持后续维护和新项目。

**总结会议流程**

1. 会前收集数据：计划、进度、成本、缺陷、变更、风险、验收结果、用户反馈。
2. 会议回顾目标和成果。
3. 分析偏差和问题原因。
4. 总结优秀实践和经验教训。
5. 形成改进建议和行动项。

**会议注意事项**

- 基于事实和数据，而不是个人情绪。
- 关注机制和过程改进，避免变成追责会。
- 会议结束后要形成可执行的总结报告和改进行动。

**总结报告内容**

| 模块 | 内容 |
| --- | --- |
| 项目概况 | 背景、目标、范围、周期、团队 |
| 完成情况 | 范围、进度、成本、质量目标达成 |
| 管理分析 | 风险、变更、沟通、团队协作 |
| 经验教训 | 成功做法、主要问题、根本原因 |
| 改进建议 | 后续项目可复用措施和制度改进 |
| 后续行动 | 维护交接、遗留问题、责任人和期限 |

**易考点**

- 项目总结不是形式文档，而是组织学习。
- 好的总结要能转化为模板、规范、检查表或改进行动。

<a id="chapter-summary"></a>
## 3. 本章总结

本章强调项目最后阶段的正式确认和组织学习。验收保证项目成果被客户接受，收尾保证责任、文档、资源和合同事项闭合，总结改进保证经验进入下一次项目。考试中重点掌握验收前提、验收测试内容、验收流程、验收报告和总结会议组织方法。

<a id="key-points"></a>
## 4. 本章重点内容

- 收尾包括验收、移交、归档、结算、资源释放、总结改进。
- 验收前提：交付物完成、测试通过、关键缺陷关闭、文档齐全、用户培训、标准明确。
- 验收测试：功能、流程、数据、性能、安全、兼容、易用、文档。
- 验收流程：申请、准备、测试、记录、整改复验、结论、报告签署。
- 项目总结目的：评估目标、沉淀经验、发现问题、改进过程、支持维护。
- 总结会议要基于数据和事实，避免追责，形成行动项。

<a id="exercises"></a>
## 5. 习题及答案+解析

1. 项目经理在项目收尾工作中起到什么作用？

   答案：项目经理在收尾中起组织、协调、确认和总结作用。要确认交付物完成，包括软件系统、源代码、部署文档、用户手册、测试报告和培训材料；组织验收测试和验收会议；协调处理遗留问题；推动文档归档、资源释放、合同结算和运维交接；组织项目总结，分析目标、进度、成本、质量、风险和经验教训。

   解析：项目经理不是最后签个字，而是确保项目被正式接受、责任完成、知识沉淀。

2. 项目验收的前提是什么？

   答案：前提是交付物达到约定验收条件，具备可检查、可运行、可交付状态。包括需求范围已实现或经双方确认调整；关键缺陷已修复，遗留缺陷有处理方案；系统测试、集成测试及必要性能、安全测试完成；部署环境稳定；数据迁移完成或具备试运行数据；用户手册、部署文档、测试报告、培训材料齐全；用户完成必要培训；验收标准和用例已确认。

   解析：验收前提要覆盖软件、缺陷、环境、数据、文档、培训和标准。

3. 项目验收测试应该包括哪些方面？

   答案：包括功能验收、业务流程验收、数据验收、性能验收、安全验收、兼容性验收、易用性验收和文档验收。学校教学管理系统中，功能验收覆盖登录权限、学生、教师、课程、选课、成绩、考勤和报表；流程验收模拟新学期排课、学生选课、教师录入成绩、教务导出报表；数据验收检查历史数据完整准确；安全验收验证角色权限隔离。

   解析：验收测试要从用户业务目标出发，不只是重复开发测试。

4. 项目总结的目的有哪些？

   答案：目的包括评估项目目标达成情况、沉淀经验教训、发现过程问题、表彰团队贡献、改进组织能力和支持后续维护。通过总结比较计划与实际在范围、进度、成本、质量和风险方面的差异，分析成功和失败原因，把隐性经验转化为组织知识。

   解析：项目总结的根本价值是组织学习和持续改进，不是形式化写报告。

5. 如何召开项目总结会议？

   答案：会前明确目的、范围、参与人员和材料，收集计划、进度、成本、缺陷、变更、风险、验收结果和用户反馈，并让成员准备个人总结。会议中按目标回顾、主要成果、偏差分析、问题原因、优秀实践、经验教训和改进建议展开。主持人应营造客观开放氛围，避免追责。会后形成总结报告和行动项，明确哪些经验进入模板，哪些问题需要制度改进。

   解析：总结会要基于事实和数据，输出可执行改进项。

6. 软件度量的意义是什么？结合项目尝试做度量分析，看看有什么收获？

   答案：软件度量用数据认识项目和产品状态，支持计划、控制、评价和改进。以学校教学管理系统为例，若项目 16.2 KLOC，系统测试发现 81 个缺陷，则缺陷密度为 `81 / 16.2 = 5 个/KLOC`；若成绩管理模块发现 25 个缺陷，说明该模块业务规则复杂，应加强评审和测试。若平均缺陷修复周期为 3 天、严重缺陷为 6 天，说明严重缺陷需要更高优先级。若需求变更率达到 20%，说明前期需求确认不足。

   解析：度量题要体现“数据-分析-改进”。数字本身不是目的，关键是从数据看出问题并提出改进。
