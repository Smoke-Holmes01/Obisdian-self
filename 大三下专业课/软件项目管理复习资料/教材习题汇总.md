# 软件项目管理 教材习题汇总

本篇文档汇总了《软件项目管理》教材第1章至第10章的课后习题及参考解答。

---

## 第1章 概述

1. 结合生活中某件事，谈谈项目管理的作用。

   答：以组织一次班级毕业旅行为例，如果没有项目管理，大家可能只是在群里零散讨论目的地、预算、交通和时间，最后容易出现费用超支、人员意见不统一、订票延误等问题。采用项目管理方法后，可以先明确目标，例如“在周末完成一次预算不超过每人800元的两日游”；再分解任务，如路线选择、预算统计、住宿预订、交通安排、安全预案等；然后安排负责人和截止时间，并在执行过程中持续跟踪。项目管理的作用就在于把临时性的复杂事情变成有目标、有计划、有分工、有控制的活动，使有限资源得到合理利用，减少风险，提高完成质量。

2. 软件项目与一般项目的区别在什么地方？软件项目管理中最突出的问题是什么？

   答：软件项目与一般项目的共同点是都有明确目标、资源约束、时间限制和临时性组织，但软件项目更突出无形性、复杂性和不确定性。建筑、制造等项目的成果通常可直观看到，而软件在开发过程中主要表现为需求、设计、代码、测试用例等知识性成果，进度和质量不易直接观察。软件需求还容易变化，技术方案可能不断调整，人员能力差异也会显著影响结果。因此软件项目管理中最突出的问题通常是需求、进度、成本和质量之间的协调，尤其是需求变更带来的范围蔓延、返工和延期。管理者必须通过需求管理、迭代交付、风险控制和持续沟通来降低不确定性。

3. 项目管理的要素有哪些？怎样衡量项目是否成功？

   答：项目管理的核心要素包括范围、时间、成本、质量、资源、风险、沟通和干系人等。其中范围决定“做什么”，时间决定“什么时候完成”，成本决定“花多少钱”，质量决定“做到什么标准”，资源和团队决定“由谁来做”，风险管理处理不确定事件，沟通和干系人管理保证信息一致、期望一致。衡量项目是否成功不能只看是否交付，还要看是否在约定范围内按期、按预算、按质量完成，是否满足用户和业务目标，是否让主要干系人接受，是否留下可维护的成果和可复用的经验。例如一个学校信息系统即使按期上线，如果教师不会用、数据经常错误，也不能算真正成功。

4. 如何理解在任务、进度和成本之间获得平衡？有什么具体实例可以说明？

   答：任务、进度和成本之间存在明显制约关系。任务范围扩大，通常需要更多时间或更多成本；如果进度被压缩，可能需要增加人员、加班或降低部分功能优先级；如果成本固定，则必须控制范围或接受更长工期。所谓平衡，就是在项目目标和约束之间做可接受的取舍，而不是单方面追求“功能最多、时间最短、费用最低”。以学校教学管理系统为例，原计划三个月完成学生管理、课程管理、成绩管理和报表功能。若学校临时要求增加移动端考勤，在工期不变的情况下，应评估新增工作量，选择增加开发人员、推迟低优先级报表，或把移动端考勤放入下一期，而不能简单要求团队无条件完成所有内容。

5. 收集相关资料，对 PMBOK 和 PRINCE2 进行比较，阐述各自的特点。

   答：PMBOK 是项目管理知识体系指南，强调项目管理知识领域、过程组、原则和绩效域，适合帮助项目经理建立系统化的管理知识框架。它更像一套“应当关注什么”的通用知识库，覆盖范围、进度、成本、质量、资源、沟通、风险、采购、干系人等方面，适用于多行业项目。PRINCE2 则是一种结构化项目管理方法，强调商业论证、组织角色、阶段控制、例外管理、产品导向计划和经验学习，更像一套“如何组织和控制项目”的方法论。两者区别在于：PMBOK 偏知识体系和管理能力框架，PRINCE2 偏过程、角色和治理机制。实际项目中可以结合使用：用 PMBOK 补全知识领域，用 PRINCE2 加强阶段授权、决策和控制。

## 第2章 项目准备和启动

1. 软件项目的可行性分析包括哪些方面？影响决策的关键因素又是什么？

   答：软件项目可行性分析通常包括技术可行性、经济可行性、操作可行性、进度可行性、法律合规可行性和风险可行性。技术可行性关注现有技术、平台、团队能力能否支持项目；经济可行性比较投入成本与预期收益；操作可行性判断用户是否愿意使用、流程是否能落地；进度可行性评估期限是否现实；法律合规可行性检查数据、版权、合同和安全要求；风险可行性则分析关键风险是否可控。影响决策的关键因素包括业务价值是否明确、投资回报是否合理、核心需求是否稳定、技术难点是否可突破、组织是否有资源支持以及主要干系人是否达成一致。若项目价值不清或关键风险无法控制，应谨慎启动。

2. 项目的组织结构主要分哪几类？软件项目的组织结构通常采用哪一类？

   答：项目组织结构常见类型包括职能型、项目型和矩阵型。职能型组织按部门划分，人员归属稳定，专业管理强，但项目协调效率较低；项目型组织围绕项目组建团队，项目经理权力大、响应快，但资源可能重复配置，项目结束后人员安置压力较大；矩阵型组织介于两者之间，成员既属于职能部门，也参与项目工作，可分为弱矩阵、平衡矩阵和强矩阵。软件项目通常采用矩阵型或轻量项目型结构，因为软件开发需要产品、开发、测试、运维、业务人员协同，同时又要保留技术部门的专业积累。小型敏捷团队也常采用跨职能团队形式，使需求、开发、测试和交付形成闭环。

3. 软件开发模型有哪些？各自的特点是什么？针对你的项目会选择哪个模型？

   答：常见软件开发模型包括瀑布模型、快速原型模型、增量模型、迭代模型、螺旋模型和敏捷模型。瀑布模型阶段清晰，适合需求稳定、合规要求强的项目，但对变化响应慢；快速原型模型通过原型帮助用户澄清需求，适合需求不明确的场景；增量模型分批交付功能，能较早产生价值；迭代模型在多轮开发中逐步完善系统；螺旋模型强调风险分析，适合大型高风险项目；敏捷模型强调用户参与、短周期迭代和持续反馈。针对学校教学管理系统，我会选择增量迭代结合 Scrum 的方式，先交付用户登录、学生与课程管理，再逐步交付选课、成绩、报表等功能，以便学校尽早试用并及时反馈。

4. 针对 XP 和 Scrum 两种开发模式进行比较，了解它们有什么不同和各自的优势。

   答：XP 和 Scrum 都属于敏捷方法，但关注重点不同。Scrum 更偏项目组织和过程管理，强调产品负责人、Scrum Master、开发团队三个角色，以及产品待办列表、Sprint计划、每日站会、评审会和回顾会等节奏。它的优势是框架清晰，便于管理需求优先级、迭代目标和团队协作。XP 更偏工程实践，强调结对编程、测试驱动开发、持续集成、简单设计、重构、小版本发布和代码集体所有。它的优势是能通过技术实践提高代码质量和响应变化的能力。学校教学管理系统可用 Scrum 管理迭代和需求优先级，同时借鉴 XP 的持续集成、自动化测试和重构实践，避免只敏捷开会而缺少工程质量保障。

5. 针对 BDD 和 FDD 两种开发模式进行比较，了解它们有什么不同和各自的优势。

   答：BDD 即行为驱动开发，强调用业务人员、测试人员和开发人员都能理解的行为描述来表达需求，通常采用“给定-当-那么”的场景形式。它的优势是促进需求沟通，减少开发理解偏差，使验收标准更清晰。FDD 即功能驱动开发，强调围绕功能列表进行建模、计划、设计和构建，适合把系统拆成可管理的小功能并持续交付。它的优势是功能导向明确，便于进度跟踪和责任划分。两者区别在于，BDD 更关注用户行为和验收场景，FDD 更关注功能清单和功能实现流程。学校教学管理系统中，可以用 BDD 描述“教师录入成绩后学生可查询”的行为，用 FDD 管理“成绩录入、成绩查询、成绩统计”等功能开发。

6. 用一句话概括软件项目经理的作用是什么，软件项目经理应具备哪些能力和素质？

   答：软件项目经理的作用可以概括为：在既定目标和约束下，整合人员、技术、进度、成本、质量和风险，使项目成果满足客户和组织期望。项目经理应具备计划能力、组织协调能力、沟通能力、风险识别与应对能力、质量意识、成本意识和决策能力。软件项目经理还应理解软件工程过程，能与开发、测试、业务和运维人员有效沟通，判断技术风险和需求变化影响。素质方面，应具备责任心、诚信、抗压能力、服务意识、学习能力和团队领导力。优秀项目经理不是单纯催进度的人，而是帮助团队排除障碍、平衡冲突、维持透明度并推动项目持续达成目标的人。

7. 什么是软件项目相关利益人？如何对他们进行分析和管理？

   答：软件项目相关利益人，也称干系人，是指会影响项目或受项目影响的个人、团队或组织，包括客户、最终用户、项目发起人、项目经理、开发团队、测试团队、运维人员、供应商、管理层和监管方等。干系人管理首先要识别干系人，明确他们的角色、利益诉求、权力大小和对项目的影响；其次进行分类分析，例如按权力-利益矩阵区分重点管理、保持满意、保持告知和低度关注对象；再次制定沟通和参与策略，如定期汇报、需求评审、原型演示、用户培训等。对学校信息系统而言，校领导关注管理效率，教务处关注流程准确，教师关注易用性，学生关注查询便利，项目组必须分别管理其期望。

## 第3章 项目计划

1. 为什么说项目计划不是一个文档，而是一个持续的策划过程？

   答：项目计划最终会形成文档，但计划的本质不是一次性写出文件，而是在项目全生命周期中持续分析目标、约束、资源和风险，并根据变化不断调整。软件项目尤其如此，因为需求、技术、人员和外部环境都可能变化。项目早期计划通常基于有限信息，只能形成总体安排；随着需求细化、设计完成、开发推进，估算会更准确，风险会暴露，计划也应滚动更新。如果把计划看成静态文档，项目很容易出现计划与现实脱节的问题。正确做法是把计划作为沟通、协调和控制的工具，定期检查范围、进度、成本、质量和风险，根据偏差采取纠正措施，使项目始终朝目标推进。

2. 深刻理解本章中所介绍的计划方法，然后讨论每种方法的优点和缺点。

   答：常见计划方法包括滚动计划、WBS分解和网络计划技术。滚动计划法先做总体规划，再对近期工作做详细计划，优点是适应不确定性，适合软件项目需求逐步明确的特点；缺点是对项目经理持续跟踪和调整能力要求高，若缺少纪律容易变成没有长期控制。WBS方法把项目成果和工作逐层分解，优点是能明确范围、责任和估算基础，便于进度与成本管理；缺点是分解过粗会遗漏工作，分解过细会增加管理成本。网络计划技术通过活动关系分析关键路径，优点是能识别关键活动和缓冲时间，便于进度控制；缺点是依赖较准确的活动历时估计，对频繁变化的敏捷项目需要结合迭代管理使用。

3. 在第2章《某私立中学小型信息管理系统可行性研究报告》基础之上，制定该项目的一个总体计划，包括资源、进度、风险和质量等管理计划。

   答：学校教学管理系统总体目标是在一个学期内完成学生、教师、课程、选课、成绩、考勤和统计报表等核心功能，支持教务处日常管理。资源计划方面，建议配置项目经理1人、需求/业务分析1人、后端开发2人、前端开发1人、测试1人、兼职运维1人，校方安排教务代表和教师代表参与评审。进度计划可分为启动与需求2周，概要设计和原型2周，核心模块开发6周，测试与修复3周，试运行和验收3周。风险计划重点关注需求变更、历史数据质量、用户培训不足和上线时间冲突，并分别设置变更评审、数据清洗、培训演示和灰度上线措施。质量计划包括需求评审、设计评审、代码评审、单元测试、集成测试、用户验收测试和缺陷跟踪，确保系统稳定、易用、数据准确。

4. 熟悉一个项目计划工具，然后将上述计划输入到系统内，实现计划的信息系统管理。

   答：可以选择 Microsoft Project、禅道、Jira、Trello 或飞书项目等工具管理学校教学管理系统计划。以 Microsoft Project 为例，先建立项目日历，设置工作日和节假日；再录入WBS任务，如需求调研、原型设计、数据库设计、学生管理开发、课程管理开发、成绩管理开发、系统测试、用户培训和验收上线；然后为每个任务设置工期、前置关系和负责人，形成甘特图。对于敏捷团队，也可以用 Jira 建立产品待办列表，把功能拆成用户故事，按 Sprint 安排开发和测试。使用工具的价值不只是画图，而是实现任务分派、进度跟踪、风险记录、缺陷闭环和状态可视化，使计划从静态文档变成可执行、可检查、可调整的信息系统。

## 第4章 项目估算

1. 针对学校教学管理系统，进行功能分解。以 LOC 估算每个功能的规模。假设你所在的公司平均生产率是 450LOC/pm，且平均劳动力价格是每人月 7000 美元，使用基于 LOC 的估算技术来估算构建该软件所需的工作量及成本。

   答：可将学校教学管理系统分解为登录与权限管理1200 LOC、学生信息管理1800 LOC、教师信息管理1200 LOC、课程管理1500 LOC、选课与排课2500 LOC、成绩管理2200 LOC、考勤管理1400 LOC、考试管理1200 LOC、统计报表2000 LOC、系统维护与基础配置1200 LOC，总规模约为16200 LOC，即16.2 KLOC。公司平均生产率为450 LOC/人月，因此工作量估算为：`16200 / 450 = 36 人月`。若平均劳动力价格为每人月7000美元，则人力成本为：`36 * 7000 = 252000 美元`。该结果只反映基于代码规模和生产率的人力成本，实际预算还应考虑服务器、软件工具、培训、数据迁移、管理成本和风险储备。

2. 使用 COCOMO 模型估算学校教学管理系统开发项目的工作量，并采用其他方法进行估算，然后使用三点估算法导出该项目的单一估算值。

   答：学校教学管理系统规模约为16.2 KLOC，属于较小、需求相对明确的管理信息系统，可按基本COCOMO有机型估算。公式为 `E = 2.4 * KLOC^1.05`，代入后得到 `E = 2.4 * 16.2^1.05 ≈ 44.7 人月`。若估算开发周期，可用有机型公式 `T = 2.5 * E^0.38`，约为10.6个月。其他方法中，LOC生产率法得到36人月；类比同类学校管理系统，若考虑数据迁移、报表和验收沟通，可估为52人月。使用三点估算法，取乐观值O=36、最可能值M=44.7、悲观值P=52，则单一估算值为 `(O + 4M + P) / 6 = (36 + 178.8 + 52) / 6 ≈ 44.5 人月`。对应人力成本约为 `44.5 * 7000 = 311500 美元`。

3. 成本和进度估算是在软件项目计划期间——详细的软件需求分析或设计进行之前完成的。你认为为什么会这样？是否存在不需要这样做的情况？

   答：成本和进度估算通常必须在详细需求和设计之前完成，因为项目启动、立项、投标、合同签订和资源安排都需要先判断项目是否值得做、能否做、需要多少预算和大致什么时候交付。管理层和客户不可能等到全部设计完成后才决定是否投入资源。因此早期估算虽然不精确，但可以提供决策依据，并随着项目推进逐步修正。不存在完全不需要估算的情况，只是估算粒度和形式不同。对于探索性原型、研究型项目或内部创新项目，早期可能只做时间盒和预算上限估算，而不承诺完整范围；对于敏捷项目，也会先做发布级估算，再在每个迭代中细化故事点和任务工时。也就是说估算不可避免，但可以滚动、分层和渐进。

## 第5章 项目进度和成本管理

1. 如何正确标识出软件项目活动？

   答：正确标识软件项目活动应从项目范围和WBS开始，把可交付成果逐步分解为可执行、可估算、可分派和可检查的活动。活动不宜过大，否则难以跟踪；也不宜过细，否则管理成本过高。一般应先明确项目阶段，如需求、设计、开发、测试、部署、验收；再识别每个阶段的具体活动，如需求访谈、原型评审、数据库设计、接口开发、单元测试、用户培训等。标识活动时还要考虑依赖关系、资源约束、质量活动和管理活动，不能只列编码任务。以学校教学管理系统为例，除学生管理模块开发外，还应列出数据字典确认、角色权限设计、历史数据导入、测试用例编写和验收演示等活动。

2. 各个软件项目活动之间有哪些几种依赖关系？请结合你身边的项目举例说明。

   答：项目活动之间常见依赖关系包括完成-开始、开始-开始、完成-完成和开始-完成，其中最常用的是完成-开始，即前一活动完成后后一活动才能开始。依赖来源可以分为强制依赖、选择性依赖和外部依赖。强制依赖由客观逻辑决定，例如数据库表结构未确定前，数据导入程序难以完成；选择性依赖来自团队经验，例如先做原型评审再开发界面；外部依赖来自项目外部，例如服务器采购完成后才能部署试运行。以学校教学管理系统为例，需求访谈完成后才能形成需求规格说明；课程管理模块可以与学生管理模块并行开发；系统测试必须等主要模块集成后进行；用户培训通常依赖试运行环境部署完成。

4. 什么是里程碑？如何设定里程碑？里程碑的验收标准为什么重要？结合你生活或者工作设立一个小项目（例如旅游计划和实施），试试如何设立里程碑及验收标准。

   答：里程碑是项目中的重要节点，通常代表某个阶段性成果完成或关键决策点到达。里程碑本身一般不表示持续时间，而表示“是否达到某个状态”。设定里程碑时应选择对项目控制有意义的节点，如需求确认、原型评审通过、核心功能完成、系统测试通过、用户验收完成等。验收标准重要，是因为没有清晰标准，里程碑就只是日期，无法判断成果是否真正完成。以学校教学管理系统为例，可设里程碑：第2周完成需求确认，标准是需求规格说明经教务处签字；第4周完成原型评审，标准是主要用户确认页面流程；第10周完成核心功能开发，标准是学生、课程、选课、成绩模块通过集成测试；第16周完成验收，标准是验收测试通过且遗留缺陷均有处理计划。

5. 影响软件进度和成本的因素有哪些？哪些是你完成的项目中遇到过的？请结合你的项目进行分析、找到切实可行的解决办法。

   答：影响软件进度和成本的因素包括需求变化、估算偏差、人员能力与稳定性、技术难度、沟通效率、质量问题、外部依赖、工具环境和管理成熟度。学校教学管理系统中最容易遇到的是需求变化和数据质量问题。例如教务处在开发中期提出新增按年级、班级、课程维度的复杂统计报表，会增加设计、开发和测试工作；历史学生数据格式不统一，也会拖慢数据导入。解决办法包括建立变更控制流程，要求新增需求说明影响范围、优先级和是否替换原有功能；采用迭代交付，先保证核心流程上线，再扩展报表；提前进行数据摸底和清洗；设置风险储备；通过每日站会和看板及时暴露阻塞，避免问题积累到后期。

6. 总结一下看板管理的特点和好处？

   答：看板管理的特点是把工作项可视化，通过“待办、进行中、测试中、已完成”等列展示任务状态，并限制在制品数量，推动团队按流动效率完成工作。它强调拉动式工作，即成员在有能力处理时从上游拉取任务，而不是被动堆积任务。看板的好处包括：一是透明，所有人都能看到当前任务、负责人和阻塞；二是便于发现瓶颈，例如测试列长期堆积说明测试资源不足；三是能减少多任务切换，通过限制进行中任务提高完成率；四是适合持续交付和运维类工作。学校教学管理系统开发中，可把需求、开发、代码评审、测试、待发布、已完成可视化，项目经理据此跟踪进度并协调资源。

7. 什么是挣值管理？通过一个具体项目来应用这种方法。

   答：挣值管理是一种综合范围、进度和成本的项目控制方法，核心指标包括计划价值PV、挣值EV和实际成本AC。PV表示按计划到某时点应完成工作的预算价值，EV表示实际已完成工作的预算价值，AC表示实际花费。由此可计算进度偏差 `SV = EV - PV`、成本偏差 `CV = EV - AC`、进度绩效指数 `SPI = EV / PV`、成本绩效指数 `CPI = EV / AC`。例如学校教学管理系统预算为30万元，计划第8周完成50%工作，则PV为15万元；实际完成40%，EV为12万元；实际花费14万元，AC为14万元。则SV=-3万元，说明进度落后；CV=-2万元，说明成本超支；SPI=0.8，CPI=0.86。项目经理应分析落后原因，调整范围、资源或迭代计划。

## 第6章 项目质量管理

1. 谈谈你是如何认识软件质量的。

   答：软件质量不仅是“程序能运行”，而是软件满足明确需求和隐含期望的程度。它既包括功能正确性，也包括可靠性、易用性、性能、安全性、可维护性、兼容性和可扩展性。对用户而言，质量表现为系统好用、稳定、数据准确、响应及时；对开发团队而言，质量还表现为代码结构清晰、缺陷少、易修改、易测试；对组织而言，质量意味着系统能支撑业务目标并降低长期维护成本。软件质量不是测试阶段才产生的，而是在需求、设计、编码、评审、测试、部署和运维全过程中形成的。学校教学管理系统如果成绩计算错误，即使界面美观也不能算高质量；如果功能正确但操作复杂，也会影响质量评价。

2. 如果某个项目是开发一个学校信息系统，如何完成该项目的质量计划？

   答：学校信息系统质量计划应先明确质量目标，例如核心业务功能正确率高、关键页面响应时间小于3秒、重要数据零丢失、严重缺陷上线前全部关闭、用户验收通过。其次确定质量标准和评审活动，包括需求评审、原型评审、数据库设计评审、接口评审和代码评审。再次制定测试策略，包括单元测试、集成测试、系统测试、性能测试、安全测试、兼容性测试和用户验收测试，重点覆盖登录权限、学生信息、课程选课、成绩录入和报表统计等核心流程。还要规定缺陷管理流程，明确缺陷等级、修复时限、回归测试和关闭标准。最后安排质量角色和度量指标，如缺陷密度、测试通过率、评审问题数、返工率和用户满意度，并在项目例会上持续跟踪。

3. 请简要说明评审的基本流程。

   答：软件评审的基本流程通常包括计划、准备、召开评审会议、记录问题、修改跟踪和关闭。首先由负责人确定评审对象、目的、范围、参与人员和时间，例如评审需求规格说明或数据库设计。其次将材料提前发给评审人员，评审人员独立阅读并记录问题。评审会议中，作者介绍背景，评审人员围绕完整性、正确性、一致性、可行性和可测试性提出问题，主持人控制节奏，记录员记录缺陷和改进项。会议后，责任人按问题清单修改文档或代码，并说明处理结果。最后由主持人或质量人员检查问题是否关闭，必要时组织复审。评审重点不是追责，而是尽早发现问题，降低后续返工成本。

4. 谈谈缺陷趋势分析和分布分析有什么不同，它们对质量管理工作有什么帮助？

   答：缺陷趋势分析关注缺陷数量随时间或阶段的变化，例如每天新增缺陷、关闭缺陷、遗留缺陷的趋势。它能帮助项目判断质量是否趋于稳定、测试是否充分、修复是否及时。如果系统测试后期新增严重缺陷仍持续上升，说明质量风险较高，不宜匆忙上线。缺陷分布分析关注缺陷在模块、类型、严重级别、来源阶段或责任原因上的分布，例如成绩管理模块缺陷最多，或界面类缺陷占比最高。它能帮助团队找到质量薄弱点和根本原因。两者结合使用更有效：趋势分析回答“质量是否在变好”，分布分析回答“问题主要在哪里”。学校信息系统中若报表模块缺陷集中，应增加专项测试和设计复查。

5. 有哪些指标可以用来测量软件过程质量？

   答：软件过程质量指标用于衡量开发过程是否稳定、有效和可改进。常见指标包括需求变更率、需求评审缺陷数、设计评审问题关闭率、代码评审覆盖率、单元测试覆盖率、构建成功率、缺陷移除率、缺陷泄漏率、返工率、测试用例执行率、缺陷修复周期、迭代计划完成率和过程遵从率等。过程质量不同于产品质量，它关注“如何做”的质量。例如缺陷泄漏率高，说明早期评审和测试没有充分发现问题；构建失败频繁，说明集成过程不稳定；需求变更率过高，说明需求沟通或范围控制存在问题。学校教学管理系统可重点跟踪需求变更率、评审问题关闭率、缺陷修复周期和验收缺陷数，用于改进项目过程。

6. 试通过鱼骨图对目前项目中的问题进行分析，找出其根本原因。

   答：以学校教学管理系统“系统测试阶段缺陷较多、进度延误”为问题，可以从人员、方法、工具、需求、技术和管理六个方面做鱼骨分析。人员方面，部分开发人员对教务业务不熟，测试人员介入较晚；方法方面，需求评审不充分，接口约定不清，单元测试不足；工具方面，缺少持续集成和自动化回归测试；需求方面，教务处多次新增统计口径，验收标准不够明确；技术方面，历史数据格式复杂，权限模型设计不足；管理方面，风险暴露不及时，任务缓冲不足。综合分析，根本原因不是单纯开发速度慢，而是需求确认和质量前移不足。改进措施应包括加强需求和原型评审、明确验收标准、建立接口规范、提前准备测试用例和数据清洗方案。

## 第7章 项目风险管理

1. 风险管理中，最重要的事情是什么？最难处理的问题是什么？

   答：风险管理中最重要的是提前识别和主动应对，而不是等问题发生后被动救火。风险具有不确定性，但很多风险在项目早期已有迹象，如需求频繁变化、关键人员不可替代、技术方案未验证、外部接口依赖不明确等。项目经理应建立风险清单，评估概率和影响，确定责任人、应对策略和触发条件，并在项目例会上持续跟踪。最难处理的问题通常是人的认知和组织行为：有些团队不愿暴露风险，担心被追责；有些干系人低估风险，要求不现实承诺；还有些风险涉及外部环境，项目组控制能力有限。因此有效风险管理需要透明文化、数据支持和管理层参与。

2. 需求变更的风险如何防范和控制？

   答：防范需求变更风险首先要在项目前期做好需求获取和确认，通过访谈、原型、用户故事、验收标准和需求评审减少理解偏差。其次要建立需求基线，明确哪些需求已确认，后续变更必须走变更控制流程。控制变更时，应要求提出人说明变更原因、业务价值和期望时间，项目组评估对范围、进度、成本、质量和风险的影响，再由变更控制委员会或项目负责人决策。对低价值、高影响的变更应推迟或拒绝；对必须变更的内容，应调整计划和合同。学校教学管理系统中，如果新增“家长端成绩查询”，应评估权限、安全、界面和测试影响，不能直接插入当前迭代，避免核心上线目标失控。

3. 在软件开发过程中，严格执行开发流程和规范，对防范风险有什么帮助？

   答：严格执行开发流程和规范可以把很多隐性风险显性化、前置化和可控制化。需求评审能减少需求误解风险；设计评审能发现架构和接口缺陷；编码规范能降低维护风险；代码评审能发现逻辑错误和安全问题；版本管理规范能避免代码丢失和覆盖；测试流程能降低缺陷泄漏风险；发布流程能减少上线事故。流程和规范的价值不是增加文档负担，而是让团队在复杂协作中保持一致标准。学校教学管理系统涉及学生、教师、成绩等敏感数据，如果没有权限设计规范和测试流程，可能出现越权访问或成绩误改等严重问题。严格规范可以提高可追溯性，使问题发生时能快速定位和纠正。

4. 如何更好地应用 VERT 技术？

   答：VERT 技术是一种将风险、时间和成本结合起来进行网络分析的方法，适合用于不确定性较强、活动结果可能分支的项目。要更好地应用VERT，首先要建立清晰的项目活动网络，识别关键活动、可选路径和可能结果；其次要为活动持续时间、成本和成功概率设置合理估计，而不是只给单一确定值；再次要结合历史数据、专家判断和敏感性分析，找出对项目结果影响最大的风险活动；最后要根据分析结果制定应对策略，如增加验证原型、设置缓冲、准备替代方案。学校教学管理系统中，可对“历史数据迁移”建立乐观、正常、失败重做等分支，计算不同路径对工期和成本的影响，从而提前安排数据清洗和试迁移。

5. 谈谈关键链技术和关键路径技术的异同。

   答：关键路径技术和关键链技术都用于项目进度管理，都关注哪些活动决定项目总工期，并帮助项目经理识别重点控制对象。不同点在于，关键路径技术主要基于活动逻辑关系和持续时间计算最长路径，假设资源能够按计划满足；关键链技术在关键路径基础上进一步考虑资源约束和人的行为因素，如多任务切换、学生综合征、帕金森定律等。关键链通常会压缩单个任务中的安全时间，把安全时间集中为项目缓冲、接驳缓冲和资源缓冲，以便整体控制。关键路径适合逻辑关系明确、资源冲突较少的计划分析；关键链适合资源有限、多个项目共享人员、任务容易被拖延的环境。软件项目中两者可以结合：先算关键路径，再检查资源冲突并设置缓冲。

## 第8章 项目团队与干系人

1. 团队建设有几个核心内容？简单阐述一下。

   答：团队建设的核心内容包括制度建立与执行、目标和分工管理、工作氛围、激励和过程管理。制度建立与执行是团队协作的基础，包括会议规则、代码规范、提交规范、缺陷流程等；目标和分工管理保证每个人知道项目目标、个人职责和交付标准；良好的工作氛围能提高信任和沟通效率，减少推诿；激励包括物质激励和精神激励，如认可、成长机会、授权和成就感；过程管理则通过计划、跟踪、反馈和改进保证团队持续有效运转。软件项目需要高度协作，团队建设不能只靠临时动员，而要通过稳定规则、清晰目标和持续反馈形成战斗力。

2. 思考一下自己在哪个需求层次，如何才能激励自己？

   答：如果以马斯洛需求层次理解学生或项目成员的状态，我可能处于尊重需求和自我实现需求之间。基本学习和生活条件已经满足，接下来更关注能力提升、成果被认可和个人价值实现。激励自己的方法不能只依赖外部奖励，而应把大目标拆成可完成的小目标。例如在学校教学管理系统课程项目中，可以把“完成一个系统”拆成完成需求分析、画出原型、实现登录模块、完成测试报告等阶段目标，每完成一个阶段就进行复盘和总结。同时主动承担有挑战的任务，如数据库设计或项目计划编制，通过解决问题获得成就感。还可以寻找同伴监督和老师反馈，使学习过程更有方向和持续动力。

3. 在软件开发过程中，信息和知识是如何传递的？

   答：软件开发中的信息和知识通过正式和非正式两类渠道传递。正式渠道包括需求规格说明、设计文档、接口文档、会议纪要、任务看板、缺陷单、测试报告、代码注释和知识库等，优点是可追溯、便于沉淀。非正式渠道包括面对面沟通、即时消息、结对编程、代码评审讨论、站会交流和经验分享，优点是效率高、能快速澄清问题。知识也分为显性知识和隐性知识，显性知识可写入文档，隐性知识如业务经验、架构取舍和调试技巧需要通过协作和实践传递。学校教学管理系统中，教务规则应沉淀为需求和验收标准，开发经验可通过代码评审和复盘会议共享，避免知识只掌握在少数人手中。

4. 请结合身边的事情举例说明沟通管理的重要性。

   答：沟通管理的重要性在于确保相关人员对目标、范围、责任、进度和问题有共同理解。以学校教学管理系统为例，教务处可能认为“成绩管理”包括录入、修改、审核、排名和导出，而开发团队可能只理解为录入和查询。如果前期没有充分沟通，系统开发完成后就会出现用户不满意和大量返工。再如教师希望成绩录入界面简单，管理层希望报表维度丰富，学生希望查询方便，这些期望如果不协调，就会相互冲突。有效沟通应包括需求访谈、原型演示、评审确认、会议纪要和变更记录。沟通不是简单传话，而是确认信息被正确理解，并让决策、责任和后续行动清晰可追踪。

5. 文中总结了有效沟通的几个好习惯，请结合实际生活中的案例再总结出几点。

   答：有效沟通的好习惯包括提前明确沟通目的、使用对方能理解的语言、重要事项书面确认、倾听后再回应、及时反馈和避免情绪化表达。结合实际生活，班级小组作业中常见问题是有人只在群里说“我来做一部分”，但没有说明具体内容和完成时间，最后容易重复劳动或遗漏任务。更好的做法是沟通后形成清单：谁负责资料收集，谁负责PPT，谁负责演讲，什么时候提交初稿。另一个习惯是对分歧先确认事实，再讨论观点。例如对项目延期，不应先指责某个人，而应先看任务量、阻塞点和资源情况。有效沟通的核心是减少误解、推动行动，而不是增加消息数量。

6. 双赢的概念是什么？

   答：双赢是指在合作、谈判或冲突处理中，不把对方视为必须被压倒的一方，而是通过理解双方真实利益，寻找能使双方都获得合理收益的方案。双赢并不等于双方都完全满足所有要求，而是在约束条件下实现总体利益最大化，并维护长期合作关系。在软件项目中，客户希望功能更多、交付更快、成本更低，开发团队希望需求稳定、时间合理、质量可控。如果双方坚持单方面立场，就会形成冲突；如果采用双赢思路，可以通过优先级排序、分阶段交付、核心功能先上线、低优先级需求后续迭代等方式，让客户尽早获得价值，也让团队保持可持续开发。双赢强调共同目标、透明沟通和合理让步。

7. 绩效管理是怎样的循环过程？

   答：绩效管理是一个持续循环过程，通常包括绩效计划、绩效执行与辅导、绩效评估、绩效反馈和绩效改进。首先，团队和个人要明确目标、职责、评价指标和完成标准；其次，在执行过程中管理者要持续观察进展，提供资源、指导和反馈，而不是等到期末才评价；然后根据事实和数据进行绩效评估，如任务完成质量、缺陷率、协作表现、学习成长等；评估后要进行面对面反馈，肯定成绩，指出不足；最后制定改进计划，把经验带入下一轮工作。软件团队绩效管理不能只看代码行数或加班时间，而应关注交付价值、质量、协作和持续改进。这样绩效管理才是促进成长的机制，而不是单纯考核。

8. 组织你的团队讨论一下绩效考核的有效方法并尝试去实践。

   答：软件团队绩效考核应采用多维度方法，避免单一指标导致行为扭曲。对学校教学管理系统小组项目，可以从结果、过程、质量、协作和成长五个方面考核。结果指标包括按时完成的用户故事数、关键里程碑达成情况；过程指标包括任务看板更新及时性、会议参与、文档提交；质量指标包括缺陷数量、代码评审问题、测试通过率；协作指标包括是否主动帮助他人、沟通是否清晰、是否承担公共任务；成长指标包括学习新技术、复盘改进和知识分享。实践时可采用个人自评、组内互评和项目负责人评价结合的方式，每两周进行一次轻量反馈。考核结果应主要用于改进分工和能力提升，而不是制造成员之间的对立。

9. 干系人管理的重要性是什么？如何有效地做好干系人管理？

   答：干系人管理重要，是因为项目成功不仅取决于团队是否完成软件，还取决于关键人员是否支持、接受并使用项目成果。干系人需求不一致或期望失控，会导致需求变更、验收困难和上线阻力。有效干系人管理应先识别所有干系人，包括发起人、业务部门、最终用户、技术团队、运维人员和管理层；再分析他们的权力、利益、态度和影响；然后制定差异化沟通策略。对高权力高利益者要重点管理，定期汇报并参与关键决策；对最终用户要通过访谈、原型和培训增强参与；对运维人员要提前交接部署和监控要求。学校信息系统中，教务处、教师和学生关注点不同，项目组必须分别管理期望。

## 第9章 项目监督与控制

1. 什么是项目过程度量？其方法有哪些？

   答：项目过程度量是指在项目执行过程中，对进度、成本、质量、范围、风险、资源和过程活动进行量化收集、分析和反馈，以判断项目是否按计划运行并支持改进。它关注的不只是最终结果，而是过程状态和趋势。常见方法包括直接度量、间接度量、基准比较、趋势分析、偏差分析、挣值分析、缺陷分析和统计过程控制等。直接度量如任务完成数、缺陷数、工时；间接度量如缺陷密度、生产率、测试通过率；趋势分析观察指标随时间变化；偏差分析比较计划与实际。学校教学管理系统可度量迭代完成率、需求变更数、缺陷关闭周期、测试通过率和挣值指标，用于及时发现进度落后或质量风险。

2. 过程度量的制定规则有哪些？请根据你的项目，制定一些切实可行的度量指标。

   答：制定过程度量指标应遵循目标导向、可量化、可收集、可解释、成本适当和可行动原则。指标必须服务于管理目标，不能为了统计而统计；数据来源要清晰，采集方式不能给团队带来过高负担；指标结果应能指导决策。学校教学管理系统可设置以下指标：需求变更率，用于控制范围；迭代计划完成率，用于监控进度；任务准时完成率，用于发现执行偏差；缺陷密度和严重缺陷数，用于衡量质量；缺陷平均修复时间，用于衡量响应效率；测试用例执行率和通过率，用于判断测试充分性；用户验收问题数，用于评估交付适配度；数据迁移成功率，用于控制上线风险。每周汇总并在项目例会上分析趋势。

3. 数据信息收集的方式有几种？举例说明。

   答：数据信息收集方式包括人工记录、系统自动采集、会议汇报、问卷调查、访谈、文档检查、日志分析和现场观察等。人工记录适合收集工时、问题清单和风险状态；系统自动采集适合从Git、CI、缺陷管理系统、项目管理工具中获取提交次数、构建结果、缺陷状态和任务流转时间；会议汇报适合了解进度阻塞和协作问题；问卷调查适合收集用户满意度和培训效果；访谈适合获取深层需求和使用反馈；日志分析适合发现性能和错误趋势。学校教学管理系统中，可从Jira获取任务状态，从缺陷系统获取缺陷数据，从服务器日志获取接口错误，从教师问卷获取易用性反馈。

4. 数据收集的难点在哪里？

   答：数据收集的难点主要在于数据真实性、完整性、一致性、及时性和成本控制。项目成员可能因为担心考核而美化工时或进度，导致数据失真；不同人员对缺陷严重级别、任务完成标准理解不一致，会影响数据可比性；如果数据收集依赖手工填写，容易遗漏和延迟；如果指标过多，又会增加团队负担，使成员抵触。还有一个难点是数据解释，单个指标可能被误读，例如代码提交次数多不一定代表效率高，缺陷数多也可能是测试更充分。解决办法是统一定义指标口径，尽量自动采集，减少惩罚性使用，结合多指标分析趋势，并把数据用于改进而不是简单排名。

5. 数据可视化的作用是什么？

   答：数据可视化的作用是把复杂、分散的数据转化为直观图形，帮助项目团队快速理解项目状态、发现趋势和定位问题。相对于表格数字，燃尽图、甘特图、累计流图、缺陷趋势图、看板和仪表盘更容易展示进度、质量、瓶颈和风险。可视化还能提高沟通效率，使项目经理、团队成员和干系人基于同一事实讨论问题。例如学校教学管理系统中，缺陷趋势图可以显示严重缺陷是否减少，燃尽图可以显示迭代任务是否按计划完成，看板可以显示任务是否卡在测试或评审环节。好的可视化不是追求图形复杂，而是突出关键指标、异常状态和下一步行动。

6. 结合实际项目你会选择哪个可视化方法来显示项目进度信息？

   答：对学校教学管理系统，我会结合甘特图、看板和燃尽图显示进度信息。甘特图适合向管理层展示总体计划、阶段安排、里程碑和关键依赖，例如需求确认、核心开发、系统测试、试运行和验收上线。看板适合团队日常管理，展示每个任务处于待办、开发中、代码评审、测试中、待发布或已完成状态，便于发现阻塞。燃尽图适合迭代内控制，显示剩余工作量是否按预期下降。三者侧重点不同：甘特图看全局和时间，看看板看流动和瓶颈，燃尽图看迭代节奏。实际使用时可每周更新甘特图，每天更新看板，每个Sprint查看燃尽图。

7. 简述变更控制的流程。

   答：变更控制流程通常包括提出变更、记录变更、影响分析、评审决策、实施变更、验证确认和归档。首先由干系人提交变更请求，说明变更内容、原因和期望时间；项目组将其登记到变更日志；然后分析对范围、进度、成本、质量、风险、合同和资源的影响；接着由项目经理、客户代表或变更控制委员会决定批准、拒绝、延期或要求补充信息。变更获批后，应更新需求、计划、设计、测试用例和相关文档，并安排实施。实施完成后进行测试和验收，确认变更达到预期。最后归档变更记录，保证可追溯。关键点是所有变更都必须可见、可评估、可批准，避免口头变更直接进入开发。

8. 结合你的项目，你会采取哪些有效方法来预防变更？

   答：预防变更并不是禁止所有变化，而是减少不必要、低价值和因理解偏差产生的变更。学校教学管理系统中，我会采取以下方法：第一，前期充分访谈教务处、教师、学生和管理员，覆盖主要使用场景；第二，用原型演示关键流程，让用户在开发前确认界面和操作；第三，为每个需求写清业务规则和验收标准，例如成绩修改是否需要审核；第四，建立需求优先级，先锁定核心上线范围；第五，需求评审后形成基线，后续变更必须评估影响；第六，采用迭代交付，让用户尽早试用并反馈，避免到最后集中爆发变更；第七，保留合理缓冲和后续版本计划，把非紧急需求放入下一期。

## 第10章 项目收尾

1. 项目经理在项目收尾工作中起到什么作用？

   答：项目经理在项目收尾中起组织、协调、确认和总结作用。首先，项目经理要确认合同或计划范围内的交付物是否完成，包括软件系统、源代码、部署文档、用户手册、测试报告和培训材料。其次，要组织验收测试和验收会议，协调客户、用户、开发、测试和运维人员处理遗留问题。再次，要推动项目文档归档、资源释放、合同结算和运维交接，保证项目成果能持续运行。最后，项目经理要组织项目总结，分析目标达成情况、进度成本偏差、质量问题、风险处理和经验教训。收尾不是简单宣布结束，而是确保项目被正式接受、责任完成、知识沉淀，并为后续运维和新项目提供基础。

2. 项目验收的前提是什么？

   答：项目验收的前提是项目交付物已经达到约定的验收条件，并具备可检查、可运行、可交付的状态。具体包括：需求范围已经实现或经双方确认调整；关键缺陷已修复，遗留缺陷有明确处理方案；系统测试、集成测试和必要的性能、安全测试已完成；部署环境稳定，数据迁移完成或具备试运行数据；用户手册、安装部署文档、测试报告、培训材料等文档齐全；用户代表已完成必要培训；验收标准和验收用例已得到确认。学校教学管理系统中，若成绩管理、选课和报表等核心流程尚未通过测试，或教师尚不会操作系统，就不应进入正式验收，否则验收会变成问题暴露会。

3. 项目验收测试应该包括哪些方面？

   答：项目验收测试应围绕用户业务目标和合同要求展开，主要包括功能验收、业务流程验收、数据验收、性能验收、安全验收、兼容性验收、易用性验收和文档验收。学校教学管理系统中，功能验收应覆盖登录权限、学生管理、教师管理、课程管理、选课排课、成绩录入、考勤和统计报表；业务流程验收应模拟真实教务场景，如新学期排课、学生选课、教师录入成绩、教务处导出报表；数据验收应检查历史数据导入是否完整准确；性能验收应检查多用户查询和报表生成速度；安全验收应验证不同角色权限隔离；文档验收应检查用户手册和运维手册是否可用。验收测试结果应形成正式记录，作为签署验收报告的依据。

4. 项目总结的目的有哪些？

   答：项目总结的目的包括评估项目目标达成情况、沉淀经验教训、发现过程问题、表彰团队贡献、改进组织能力和支持后续维护。通过总结，可以比较计划与实际在范围、进度、成本、质量和风险方面的差异，分析成功原因和失败原因。项目总结还可以把隐性经验转化为组织知识，例如哪些需求沟通方法有效、哪些估算偏差较大、哪些风险应提前处理。对学校教学管理系统而言，总结可以发现需求原型评审对减少返工很有效，也可能发现数据迁移准备不足导致上线延期。项目总结不是形式化写报告，而是为了让团队和组织在下一次项目中做得更好。

5. 如何召开项目总结会议？

   答：召开项目总结会议应提前明确目的、范围、参与人员和材料。会前，项目经理应收集项目计划、进度数据、成本数据、缺陷统计、变更记录、风险记录、验收结果和用户反馈，并让团队成员准备个人总结。会议中可按项目目标回顾、主要成果、偏差分析、问题原因、优秀实践、经验教训和改进建议的顺序进行。主持人应营造客观、开放的氛围，避免把总结会变成追责会。对问题要追溯到过程和机制，而不是停留在个人抱怨。会议结束后，应形成总结报告和行动项，明确哪些经验进入模板，哪些问题需要制度改进。学校信息系统项目还应邀请教务代表和运维人员参加，以获得用户和运维视角。

6. 软件度量的意义是什么？结合手边的项目，尝试做一下度量分析，看看有什么收获？

   答：软件度量的意义在于用数据认识项目和产品状态，支持计划、控制、评价和改进。没有度量，项目管理容易依赖主观感觉；有了度量，团队可以发现趋势、验证问题和评估改进效果。以学校教学管理系统为例，假设项目共完成16.2 KLOC，系统测试发现81个缺陷，则缺陷密度为 `81 / 16.2 = 5 个/KLOC`；其中成绩管理模块发现25个缺陷，占比最高，说明该模块业务规则复杂，应加强评审和测试。若平均缺陷修复周期为3天，但严重缺陷为6天，说明严重缺陷需要更高优先级和资源保障。若需求变更率达到20%，说明前期需求确认不足。通过这些度量，可以得到收获：质量问题集中在复杂业务模块，需求和评审应前移，后续项目要更早制定验收标准和测试用例。
